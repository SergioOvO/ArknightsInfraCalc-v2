# 排班改版逻辑设计（公孙长乐评审稿）

> 本文只讨论基建排班逻辑，不讨论程序如何实现。
>
> 评审目标：明确不同基建使用逻辑的边界，先完成共用能力与定时换班逻辑下的排班模型；禁止把定时换班与 Mower 动态换班混用为同一个求解器。

## 一、核心结论：不同使用逻辑必须分层

基建使用逻辑不是单一问题。常见使用逻辑至少分为三类：

1. **自动轮换**：依赖游戏内置自动轮换功能；
2. **定时换班**：代表为 MAA 与人工手动换班；
3. **动态换班**：本文中直接等同于 Mower 逻辑。

三类逻辑之间共享：

- 干员池；
- 基础游戏机制；
- 单房间 / 单组合效率计算；
- 已确认的组合合法性；
- 公孙确认的组合优先级；
- 心情消耗与最长可用时间估算能力。

但它们不共享：

- 换班触发条件；
- 替补关系设计；
- 宿舍与宿管规则；
- 菲亚梅塔使用范围；
- 是否存在班次；
- 是否允许回满、耗尽、红脸 0 心情上班；
- 输出格式与执行责任。

后续开发的第一目标是：

```text
先完成共用能力 + 定时换班逻辑下的实现。
```

其中 MAA 与人工手动换班同属定时换班，可以使用统一求解器；Mower 动态换班应单独设计，不能与定时换班混用求解。

## 二、三类基建使用逻辑

### 1. 自动轮换

自动轮换依赖游戏内置的自动轮换功能进行换班。

触发条件是：

```text
某个组合出现心情耗尽
```

触发后：

- 仅替换耗尽的组合；
- 未耗尽的组合不进行换班；
- 放上当前可以使用的替补组合；
- 若无替补组合，或替补组合正在被使用，则无法完成换班。

该逻辑要求每个主力组合尽量都有其独立替补角色。

由于不同设施、不同组合的心情消耗速度不同，尤其是中枢、制造 / 贸易、发电之间速率不同，自动轮换应尽量少使用跨设施组合，尤其应减少占用中枢的跨设施组合。

允许的跨设施关系主要是：

- 工作设施 + 宿舍挂件类组合，例如深巡 + 乌尔比安；
- 此时乌尔比安不需要换班，只需要执行深巡的换班；
- 最大允许类似自动化组合这种制造 + 发电同上同下体系。

自动轮换逻辑下：

- 一般不使用菲亚梅塔；
- 中枢空位可以放心情减免类角色；
- 不存在“班次”设计；
- 不要求宿舍精细设计；
- 只要求能正常休息、确保有人可以用。

自动轮换明确不作为 MAA / Mower 的输出，只服务游戏内置自动轮换逻辑，支持正常进行求解。

### 2. 定时换班

定时换班代表为：

- MAA；
- 人工手动换班。

二者共享同一套定时换班规则，可以使用统一求解器。

区别是：

- 人工换班受生理作息限制，每天操作频次通常最多 2-3 次；
- 人工换班对完整周期非 24h 的情况适应性更高；
- MAA 更适合 24h 周期，明确每个班次具体执行时间；
- MAA 也能兼容非 24h 周期但规律的轮流换班。

定时换班需要完整的从组合到班次设计执行。该逻辑下真正存在：

- 二班；
- 三班；
- 四班；
- 固定换班点；
- 每个班次的完整房间安排。

定时换班逻辑下：

- 宿舍中对心情消耗回复的要求不高；
- 除特殊消耗异常角色外，宿舍无需专门设计复杂“单回 / 群回”规则；
- 条件允许时，宿舍中塞 1-2 名宿管干员即可，优先群回；
- 通常由玩家声明需要的班次时长；
- 根据时长放弃存在心情消耗异常的角色或组合；
- 可以正常使用菲亚梅塔和无人机；
- 菲亚梅塔完全遵循 `docs/Fiammetta.md` 中针对定时换班的规则；
- 无人机完全遵循 `docs/DRONES.md` 中的规则；
- 一般不存在回满、耗尽、红脸 0 心情上班的设计要求。

本文后续关于“排班表实现”的讨论，默认指**定时换班逻辑**。

### 3. 动态换班（Mower）

动态换班在本文中直接等同于 Mower 逻辑。

该逻辑不是设计固定班次，而是声明：

- 主力组合；
- 替补组合；
- 替换关系；
- 菲亚梅塔使用对象；
- 宿舍配置规则。

具体何时换班、当前使用哪个组合、谁去宿舍，由 Mower 根据真实心情监控和当前状态决定。

动态换班逻辑下：

- 不存在班次设计；
- 心情监控和消耗由 Mower 完成；
- 本工具最多计算心情消耗和最长可用时间；
- 本工具应确保声明主力和替补关系时，其关系在理论上可实现；
- 替补的工作时长需要覆盖其替换的所有主力组合的休息时长总和；
- 动态换班偏好多个组合共用一个替补组合，例如 2-3 个组合共用一个替补组合。

Mower 宿舍设计上：

- 强制要求每个宿舍放一名群回、一名单回宿管；
- 具体谁来休息、谁来工作由 Mower 决策安排；
- 存在回满、耗尽、红脸 0 心情上班、宿舍黑名单等具体设计。

Mower 下菲亚梅塔的使用范围与定时换班截然不同：

- 不套用 `docs/Fiammetta.md` 中面向定时换班的但书 / 龙巫 / 清流 / 可露希尔优先级；
- 明确不会给巫恋、龙舌兰、但书、可露希尔；
- 具体 Mower 菲亚梅塔目标待后续开发 Mower 逻辑时再确认。

Mower 动态换班的排班模型暂且空余，本文不展开实现细节。

## 三、共用能力

三类逻辑虽然不能混成一个求解器，但可以共用基础能力。

### 1. 干员池与练度

共用：

- `OperBox`；
- 干员拥有情况；
- 精英化等级；
- 技能与机制数据。

### 2. 单房间效率与组合合法性

共用：

- 贸易站单房间效率；
- 制造站单房间效率；
- 发电站充能速度；
- 控制中枢全局注入；
- 同房互斥；
- shortcut / L2 / L3 机制；
- 公孙确认的体系优先级。

### 3. 心情消耗与最长可用时间

共用：

- 理论心情消耗；
- 单次从满心情开始的最长可工作时间；
- 特殊高消耗角色提示；
- 是否适合某个定时班次；
- 是否可被 Mower 主替关系覆盖。

但这些数据在不同逻辑中用途不同：

| 使用逻辑 | 心情数据用途 |
|---|---|
| 自动轮换 | 判断组合何时可能触发游戏内自动轮换 |
| 定时换班 | 判断班次时长是否安全，是否应放弃高消耗组合 |
| 动态换班 | 判断主力 / 替补关系是否可实现 |

### 4. 优化目标

对于所有基建用法，通常默认都是：

```text
峰值优先
```

也就是把主力角色放到最先用的班次。

长期平均产出可以作为后续高级模式，但不能用匿名权重混入默认规则；若未来支持，应在具体使用逻辑内部单独命名和解释。

## 四、定时换班排班模型（当前优先实现）

定时换班是当前优先完成的排班模型，适用于 MAA 与人工手动换班。

其生成顺序调整为：

```text
主力编制 → 最小换班组 → 菲亚梅塔 → 排班方案 → 宿舍与宿管
```

### 第一步：主力编制

主力编制是账号当前能够组成的最高效率工作状态，包含全部生产与功能设施。

原则：

- 默认峰值优先；
- 主力角色放到最先用的班次；
- 主力先独立确定；
- 不因替补弱、宿舍不足而直接降低主力效率；
- 若某个主力组合无法支撑用户声明的班次时长，则应提示放弃该组合或调整班次，而不是混入动态换班逻辑。

### 第二步：最小换班组

换班组指必须同时上下班的干员集合。

定时换班中仍需要识别最小换班组，但用途是为了生成完整班次，而不是交给 Mower 动态触发。

只绑定真正不能拆开的关系：

- 同站组合拆开后核心机制失效；
- 跨设施体系链要求成员同时在岗；
- 某名干员上下班会改变其他成员的效率或心情消耗；
- 暖机、心情区间或派系计数要求同步处理。

没有强依赖的干员应允许分布到不同班次或独立替换。

### 第三步：菲亚梅塔

定时换班正常使用菲亚梅塔。

菲亚梅塔在定时换班中的作用是：

```text
让某个高收益核心多上一班 / 持续上班
```

其规则完全遵循 `docs/Fiammetta.md`：

- 菲亚梅塔本人满心情时才可使用；
- 固定每小时回复 2 点心情；
- 默认换班前使用，即 `order = pre`；
- 目标选择看收益率；
- 使用后菲亚梅塔进入宿舍休息，并强制在宿舍一号位；
- 不写完整宿舍恢复模拟和全局心情最优。

定时换班下的常规目标优先级为：

```text
但书（二发电布局，1级或者2级贸易站）
> 巫恋 + 龙舌兰（三级贸易站）
> 但书（三级贸易站）
> 清流（三个贸易站布局）
> 可露希尔
```

该优先级只适用于 MAA / 人工定时换班，不适用于 Mower。

### 第四步：排班方案

根据用户声明的班次时长，生成二班、三班、四班等定时换班方案。

定时换班中的班次是真实存在的输出对象：

- 每个班次有完整房间安排；
- 每个班次有固定执行时间或相对间隔；
- MAA 需要明确每个班次具体执行时间；
- 人工换班可输出人类可读时间表，允许更贴合作息。

班次设计应避免：

- 为了“支持四班”机械生成第四套低效率编制；
- 把 Mower 的动态替补关系解释成定时换班的三班 / 四班；
- 在定时换班中引入回满、耗尽、红脸 0 心情上班等 Mower 规则。

### 第五步：宿舍与宿管

定时换班宿舍设计应保持简化。

原则：

- 宿舍主要用于下班干员正常恢复；
- 除特殊消耗异常角色外，不专门设计复杂单回 / 群回规则；
- 条件允许时放 1-2 名宿管即可；
- 宿管优先群回；
- 不默认要求每个干员回满；
- 不默认设计红脸 0 心情上班；
- 不默认设计宿舍黑名单。

## 五、自动轮换模型边界

自动轮换不是当前优先实现的 MAA / 人工排班模型，但需要在总纲中明确边界。

自动轮换输出应围绕：

- 主力组合；
- 独立替补组合；
- 哪些组合可由游戏内自动轮换处理；
- 哪些跨设施组合不适合自动轮换。

自动轮换下：

- 不设计二班 / 三班 / 四班；
- 不输出 MAA 时间表；
- 不使用 Mower 动态替换关系；
- 一般不使用菲亚梅塔；
- 宿舍只要求能正常休息。

## 六、Mower 动态换班模型边界

Mower 动态换班暂不展开完整排班模型。

后续若开发，应单独成文，至少需要确认：

- 主力组合如何声明；
- 替补组合如何声明；
- 多个主力组合共用一个替补组合的上限；
- 替补工作时长如何覆盖主力休息时长总和；
- 菲亚梅塔目标范围；
- 每个宿舍一名单回 + 一名群回的宿管优先级；
- 宿舍黑名单规则；
- 红脸 0 心情上班允许范围。

当前文档只明确边界：

```text
Mower 动态换班不得套用定时换班求解器。
```

## 七、菲亚梅塔、无人机、宿舍在不同逻辑下的适用边界

### 1. 自动轮换

- 一般不使用菲亚梅塔；
- 不做精细宿舍设计；
- 中枢空位可放心情减免角色。

### 2. 定时换班

- 可以正常使用菲亚梅塔；
- 菲亚梅塔遵循 `docs/Fiammetta.md`；
- 可以正常使用无人机；
- 无人机遵循 `docs/DRONES.md`；
- 宿舍简化设计，1-2 名宿管，优先群回；
- 适用于 MAA 与人工手动换班。

### 3. Mower 动态换班

- 不套用定时换班的菲亚梅塔目标；
- 明确不会给巫恋、龙舌兰、但书、可露希尔；
- 不设计固定班次；
- 可以正常使用无人机，规则另行开发；
- 宿舍需要每个宿舍一名单回 + 一名群回；
- 支持回满、耗尽、红脸 0 心情上班、宿舍黑名单等 Mower 特有规则。

## 八、班次概念只属于定时换班

二班、三班、四班是定时换班逻辑中的概念，主要用于 MAA 和人工手动操作。

### 二班

固定两套循环班次。

适用于：

- 用户声明需要两班逻辑；
- 具备更低的换班操作理解门槛；
- 用户愿意接受一定效率损失换取简单执行。

### 三班

固定三套循环班次。

适用于：

- 人工或MAA一天 2 次换班，12/12/12，一个周期约为36h；
- 常见 24h 周期；
- 8 / 8 / 8；
- 12 / 6 / 6；
- 其他规律三段式换班。

### 四班

四班不应机械理解为所有岗位都有三套完整替补。

只在以下情况考虑：

1. 菲亚梅塔动作、班次结构或短班需求导致需要拆分状态；
2. 深海、心情区间或其他特殊约束导致原三班被拆成更多状态；
3. MAA 或人工定时换班确实需要更细的固定执行点。

### 五班及以上

默认不支持。除非有具体账号、具体体系和明确收益，否则更多替补层大概率只会降低效率并增加执行复杂度。

### 自动轮换与动态换班

自动换班可以套用二班模型，同时约束跨设施大组合使用
Mower 动态换班不使用二班 / 三班 / 四班作为核心模型。

- 自动轮换：输出主力组合与可用替补组合；
- Mower：只有主力组合、替补组合、替换关系、实时心情状态。

## 九、本项目职责边界

### 1. 共用能力层

本项目可以共同提供：

- 干员池解析；
- 技能与机制结算；
- 单设施效率计算；
- 组合合法性判断；
- 已确认体系的优先级；
- 心情消耗与最长可用时间估算。

### 2. 定时换班层（当前优先）

本项目应输出：

- 二班 / 三班 / 四班等固定班次；
- 每个班次的完整房间安排；
- 每个班次的执行时间或相对间隔；
- 菲亚梅塔动作；
- 无人机动作；
- 简化宿舍安排；
- MAA 可用计划；
- 人工可读时间表。

### 3. 自动轮换层

后续若实现，应输出：

- 主力组合；
- 独立替补组合；
- 自动轮换可行性说明；
- 不适合自动轮换的跨设施组合提示。

### 4. Mower 动态换班层

后续若实现，应输出：

- 主力组合；
- 替补组合；
- 替换关系；
- 最长可用时间；
- 菲亚梅塔声明对象；
- 宿舍群回 / 单回配置建议；
- 宿舍黑名单等约束。

但不输出固定班次，不套用定时换班规则。

## 十、已确认事项与后续落地顺序

### A. 换班逻辑入口

已确认：

1. 用户未声明换班逻辑时，默认进入**定时换班**逻辑。
2. 后续 CLI / UI 需要明确提供模式设置：
   ```text
   auto_rotation / timed_rotation / mower_dynamic
   ```
3. Mower 动态换班作为后续模式，当前只明确边界，不进入第一阶段实现。
4. 自动轮换可视为具备特殊条件的二班制，后续在定时换班之后落地。

### B. 定时换班实现

已确认：

1. 人工定时换班默认最多支持每天 3 次操作。
2. MAA 定时换班最多建议支持 4 个班次状态。
3. MAA 非 24h 周期但规律轮换时，时间表达应采用**相对间隔**；非 24h 周期无法稳定表达为每日固定绝对时间点，但可以形成规律循环。
4. 定时换班下，理论换班时间需要安全余量；安全余量按更粗粒度取整，最好允许约半小时容错。
5. 定时换班下，如果特殊高消耗角色经过计算无法适配当前班次，则直接放弃该角色 / 组合，而不是强行设计复杂恢复或引入 Mower 逻辑。

### C. 自动轮换

已确认：

1. 自动轮换下，感知信息、人间烟火等达到跨三个设施及以上的体系应默认禁用。
2. 自动轮换下，应减少使用多名中枢参与的体系。
