本文是仓库文档生命周期的唯一规范 owner。根 `AGENTS.md`、项目 Skills、目录 README 和检查脚本只路由或执行本文，不另行定义一套状态机。

## 1. 角色

| 角色 | 职责 | 允许位置 |
|---|---|---|
| `canonical` | 裁决一个或多个稳定领域键的当前规范 | `docs/` 当前区 |
| `current-reference` | 解释当前实现、导航或工作流，不裁决业务语义 | `docs/`；精确允许的目录 README；已跟踪的 `feedback/README.md` |
| `active-change` | 记录提案、授权任务、实施状态和开放项 | `docs/TODO/` |
| `decision` | 保存日期化的架构决策、理由和替代关系 | `docs/ADR/` |
| `generated-reference` | 保存可由确定性 generator 重建的事实 | `docs/` 当前区 |
| `evidence` | 保存证据 ledger，不拥有产品工作 | 已跟踪的 `feedback/` 或具名证据路径 |
| `archive` | 保存完成、被替代或历史快照，不作为 current truth | `docs/ARCHIVE/` |

每个领域键恰好由一份 `canonical` 文档拥有。汇总、示例、实现地图和 change 只能链接 owner；不能并列定义规范性规则。

## 2. 状态

| 角色 | 合法状态 |
|---|---|
| `canonical` / `current-reference` | `current` |
| `active-change` | `proposal`、`ready-on-request`、`in-progress`、`blocked` |
| `decision` | `proposed`、`accepted`、`superseded` |
| `generated-reference` | `generated` |
| `evidence` | `current`、`closed` |
| `archive` | `completed`、`superseded`、`historical` |

`in-progress` 只表示当前有 writer，不能进入默认分支或最终 hard check。任务未在本轮完成时，交接前改为 `ready-on-request`；存在明确外部阻塞时改为 `blocked`。任务完成或被替代后不得留在 `docs/TODO/`。

## 3. 文件头元数据

受治理 Markdown 在一级标题后使用连续的可见 blockquote 键值：

```markdown
# 标题

> 文档角色：canonical
> 生命周期状态：current
> 领域键：example.domain
> 当前真源：self
> 复核触发：src/example/**
> 摘要：一句话职责
```

字段由 `scripts/codex/docs_inventory.py` 的版本化 parser 统一读取。未知键、重复键、绝对路径、仓库外路径和角色不允许字段均失败。文件头是角色、状态、领域、owner 和复核触发的唯一手工真源；不得再维护第二份手工 registry。

非 canonical 文档用 `当前真源` 链接一份或多份 current owner。`archive` 必须声明 `替代项` 或 `历史原因`；`decision` 必须声明 `当前真源`；`generated-reference` 必须声明 `生成器`。

## 4. 新鲜度与依赖

承载可变事实的 `canonical`、`current-reference`、`active-change` 和 `evidence/current` 必须声明非文档 `复核触发` 或确定性 generator。受治理 Markdown 之间通过 `当前真源` 派生依赖边，不得用 source glob 互相哈希。

检查器维护两类摘要：

- source digest：规范化排序后的非文档触发路径及 Git blob 内容。
- document digest：正文及规范元数据的语义投影；排除摘要字段、review record 和自动生成表段。

source、正文、role/status/domain、owner、trigger 或 generator 变化时，必须写入与当前摘要绑定的逐文档 review record。正文不需变化时允许 `unchanged`，但必须记录具体稳定事实和结构化 evidence reference。检查器不执行文件头中的 shell。

base 与 head 的 trigger 取并集。不能在同一变更中缩窄 trigger、删除领域键、增加 exclusion 或新增无 owner 的 source path 来绕过复核。CI hard check 缺少可验证 base 时直接失败。

## 5. 关闭事务

active change 的关闭是一个事务：

1. 通过真实入口和风险匹配证据验收。
2. 把已确认行为、限制和实现事实吸收到唯一 current owner。
3. 仍有效的开放项拆成独立 active change。
4. 从活动索引移除原 change。
5. 完成项移入 `ARCHIVE/done/`，被替代项移入 `ARCHIVE/superseded/`，历史设计移入 `ARCHIVE/plans/`。
6. 更新索引和仓库引用，并对最终树重新检查。

当有效事实已吸收、开放项已拆出、引用可同步时，Agent 必须自动执行所需的大规模移动、归档或删除。文件数和 diff 大小不是请示理由。只有业务语义冲突、产品范围变化或未吸收的用户裁决才暂停询问。

删除、移动、归档或 canonical transfer 必须由 successor、归档后的文档或迁移 ledger 携带 transition record；不存在 deleted-path 豁免。纯重复可用 hash 证明删除，其他删除必须留下“独有事实 -> current owner 章节”的映射。

transition disposition 与载体绑定：`archive-*` 只能由 `archive` 携带，`move-active-change` 只能由 `active-change` 携带，`canonical-transfer` 只能由 successor `canonical` 携带；`delete-after-absorb` 必须逐项声明 current fact mapping。

## 6. 目录规则

- `docs/TODO/` 除精确路径 `docs/TODO/README.md` 外只容纳 `active-change`。
- `docs/ADR/` 只容纳 `decision`。
- `docs/ARCHIVE/` 除精确路径 `docs/ARCHIVE/README.md` 外只容纳 `archive`。
- 根 `plans/` 不再容纳 Markdown；历史设计统一进入 `docs/ARCHIVE/plans/`。
- `feedback/` 只治理 Git 已跟踪 Markdown；ignored raw bundle 是 local-only evidence。
- `docs/INDEX.md` canonical 表和 `docs/TODO/README.md` 活动表由元数据生成并 hard check。

## 7. 失败行为

- 两份 current canonical 冲突：停止该领域迁移并请用户裁决。
- 实现不符合已确认语义：记录独立 maintenance/feature 任务，不在文档治理中顺带修代码。
- local-only evidence 不存在：标记未验证，不得宣称完整 evidence closure。
- 任一非法状态、重复/消失领域键、越界路由、活动路径历史稿、无 owner source path、摘要漂移或 transition 缺失：hard check 失败，不允许 allowlist 或 warning-only 合并。

CI 的持久门禁是全树 repository facts 与带 base 的 lifecycle continuity；它直接验证版本化文件头 attestation。`check_docs_impact.py`、`check_task_scope.py` 和 `render_evidence.py` 使用本地 task manifest，属于每次 Agent 交付必须运行的 evidence 门禁；CI 通过工具单测防止其 schema 和行为回归，但不伪造一份不存在的任务 manifest。
