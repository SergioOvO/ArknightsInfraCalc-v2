# 文档生命周期

> 文档角色：canonical
> 生命周期状态：current
> 领域键：docs.lifecycle
> 当前真源：self
> 摘要：定义文档角色、局部状态和自动归档规则

本文是仓库文档生命周期的唯一规范 owner。根 `AGENTS.md`、项目 Skills、目录 README 和检查脚本只路由或执行本文，不另行定义状态机。

## 1. 原则

- Markdown 保存当前语义、持久决策和跨会话工作；Git 保存版本、作者、时间和内容历史。
- 代码变化不会自动要求文档变化。只有行为语义、对外接口、长期决策或活动任务状态变化时，才修改最近的 owner。
- 状态变更是局部操作。关闭任务只移动任务、吸收事实并更新索引；替代 ADR 只修改新旧 ADR 和索引。
- 不在文档头保存源码摘要、文档摘要、复核记录或 source-to-document trigger，也不维护全仓 docs-impact manifest。

## 2. 角色和位置

| 角色 | 职责 | 允许位置 |
|---|---|---|
| `canonical` | 裁决一个或多个稳定领域键的当前规范 | `docs/` 当前区 |
| `current-reference` | 解释当前实现、导航或工作流 | `docs/`；目录 README；`feedback/README.md` |
| `active-change` | 保存确需跨会话继续的提案、任务和开放项 | `docs/TODO/` |
| `decision` | 保存架构决策、理由和替代关系 | `docs/ADR/` |
| `generated-reference` | 保存可由确定性 generator 重建的事实 | `docs/` 当前区 |
| `evidence` | 保存长期证据 ledger，不拥有产品工作 | 已跟踪的 `feedback/` |
| `archive` | 保存完成、被替代或历史快照 | `docs/ARCHIVE/` |

每个领域键恰好由一份 `canonical` 文档拥有。参考、示例、决策和任务通过 `当前真源` 指向 current owner，但这种链接只用于导航，不触发复核传播。

## 3. 状态

| 角色 | 合法状态 |
|---|---|
| `canonical` / `current-reference` | `current` |
| `active-change` | `proposal`、`ready-on-request`、`in-progress`、`blocked` |
| `decision` | `proposed`、`accepted`、`superseded` |
| `generated-reference` | `generated` |
| `evidence` | `current`、`closed` |
| `archive` | `completed`、`superseded`、`historical` |

`in-progress` 只表示当前有 writer，不能进入最终 hard check。任务完成或被替代后不得留在 `docs/TODO/`。

## 4. 最小文件头

受治理 Markdown 在一级标题后使用连续 blockquote 元数据：

```markdown
# 标题

> 文档角色：canonical
> 生命周期状态：current
> 领域键：example.domain
> 当前真源：self
> 摘要：一句话职责
```

所有文档只要求 `文档角色`、`生命周期状态` 和 `摘要`。其他字段按角色使用：

- `canonical`：`领域键`、`当前真源：self`。
- `current-reference`、`active-change`、`decision`、`generated-reference`、`evidence`：`当前真源`。
- `generated-reference`：`生成器`。
- `archive`：`替代项` 或 `历史原因`；可选 `快照日期`。

未知字段、重复字段、非法状态和越界路径由 `scripts/codex/docs_inventory.py` 拒绝。

## 5. 更新和关闭

当前实现或规则变化时，Agent 只更新直接拥有该事实的 canonical 或 current reference。示例、索引和其他消费者只有正文确实失真时才更新。

active change 关闭时：

1. 通过真实入口和风险匹配证据验收。
2. 把确认的行为和限制吸收到唯一 current owner。
3. 把仍有效的开放项拆成独立 active change。
4. 将原文件移入 `ARCHIVE/done/`、`ARCHIVE/superseded/` 或 `ARCHIVE/plans/`，改为 `archive` 角色和对应状态。
5. 设置 `替代项` 或 `历史原因`，运行 `docs_inventory.py --write-indexes`。
6. 检查最终树和链接。

满足上述条件时，Agent 自动完成状态修改、移动和索引更新，不因文件移动本身请求确认。语义冲突、产品范围变化或未吸收的用户裁决仍需暂停。

ADR 被替代时，只把旧 ADR 改为 `superseded`，创建或更新 successor，并维护双方正文中的明确链接。Git 历史承担其余审计记录。

## 6. 机械检查

`docs_inventory.py --check` 只检查：

- 文件头字段、角色、状态和目录位置；
- 每个领域键唯一 canonical owner；
- `当前真源` 指向现存 current 文档；
- generated reference 有具体 generator；
- 活动任务、canonical 和 ADR 生成索引无漂移；
- 最终树没有 `in-progress` 任务。

`check_repository_facts.py` 另外检查 Markdown 链接、feedback ledger 和 CLI 命令地图。检查器不判断自然语言业务语义，也不根据源码 diff 推导必须修改的文档。

## 7. 目录规则

- `docs/TODO/` 除 `README.md` 外只容纳 `active-change`。
- `docs/ADR/` 只容纳 `decision`。
- `docs/ARCHIVE/` 除 `README.md` 外只容纳 `archive`。
- 根 `plans/` 不容纳 Markdown；历史设计进入 `docs/ARCHIVE/plans/`。
- `feedback/` 只治理 Git 已跟踪 Markdown；ignored raw bundle 是 local-only evidence。
- `docs/INDEX.md` canonical / ADR 表和 `docs/TODO/README.md` 活动表由 metadata 生成。
