# 优化总结（历史归档）

> 状态：历史性能 / 建模总结。当前入口以 `AGENTS.md`、`docs/INDEX.md`、`docs/PROJECT_MAP.md` 和各专题文档为准。

全项目代码 + MECHANICS_REGISTRY.csv(727 条)已分段通读完毕。下面是面向引擎设计者的结论。

一、建模方法（v2 的本质与后续扩展套路）
引擎不是"一个解释器算一切"，而是五层叠加 + 全局资源旁路。理解这点才能正确扩展：


原文(MECHANICS_REGISTRY.csv 727条) 
   └─筛选→ prts_*_skills.json(只信此为原文)
              └─注册→ skill_table.json(buff_id→EffectAtom[]) + operator_instances.json(干员@tier→buff_ids,stepwise)
                          ↓ 运行时
   L1 interpreter   声明式积木：Phase 排序 → Selector/Condition/Action（不认识干员名，只认 buff_id）
   L2 gold_flow     有序状态机（赤金虚拟线按进驻顺序累加）—— skill_table 里 atoms:[] = 刻意委托
   L2 order_mechanic 订单分布→等效效率%（违约/裁缝peak/可露/eureka/pepe）
   L3 shortcut      组合表化最优解（巫恋核/可露分档/但书单走）+ verify 回归锚点
   ─────────
   global_resource  跨设施 producer→Pool→consumer（木天蓼/感知/人间烟火/魔物料理…16种 + CONVERSIONS）
   layout/resolve   producer 落位→注入 layout.global / global_inject→consumer 房读快照
   scenario(unit_output/schedule/assign) 产量、A-B-A 轮换、单班宏观编制
后续建模的决策树（已在 EFFECT_ATOM_DESIGN.md §8.2 定稿，代码完全贯彻）：

单 buff、可按 Phase 排序、只读 selector 写 eff/limit → L1：types.rs 加词汇 + skill_table.json 写 atoms + interpreter.rs 的 resolve_selector_value/resolve_eff_value 两处补 Selector。
进驻顺序影响后续输入（链式共享可变状态）→ L2 gold_flow 写 buff_id 分支，skill_table 留空 atoms。
订单分布/等效效率（非单笔 flat%）→ L2 order_mechanic，L1 只 TagOrder 打标。
组合已表化、是搜索/回归热路径 → L3 trade_shortcuts.json + matcher。
跨设施资源（CSV 里大量"每 N 点 X →+1% / 中枢产 X"）→ 新增 GlobalResourceKey + REGISTRY/CONVERSIONS + producer 在 control//resolve.rs 注入、consumer 用 StateConsumeToEff。
CSV 全表 727 条印证了这个分层的必要性：贸易站约 90 条、制造约 110 条是"可算效率"，其余 ~520 条是心情恢复/线索/专精速度/副产品概率——它们共享同一 EffectAtom 词汇，但本仓库刻意不建模（§8.12 非目标）。所以"完全建模"= 贸易(齐) + 制造(L1) + P0 全局资源链(木天蓼/宿舍人数)，而非 727 条全做。后续真要扩到办公室/会客室/加工站，套路就是给那个域开自己的 L2 引擎 + 一个 *_global 资源族，绝不往 L1 塞干员名分支。

三、优化（按性价比排序，全部不改变语义）
P0 — 干掉 L3 的磁盘 I/O（预计单项 10×~100× 吞吐） 把 trade_shortcuts.json 一次性加载缓存。最小侵入：shortcut.rs 用 static SHORTCUTS: OnceLock<Vec<TradeShortcutEntry>>（或 include_str! + LazyLock）；三个 matcher 改读缓存切片。更干净的做法是把 &[TradeShortcutEntry]（连同 &SkillTable）做成一个 SolveCtx/Engine 句柄，在 search_* 入口构造一次、贯穿 solve_trade_with_shift。同时给 SkillTable/instances/layout-baseline 同样待遇（进程内只 load 一次）。

P0 — 消除 owner String 与组合内深拷贝

collect_atoms 返回 (&EffectAtom, usize /*owner_idx*/) 而非 String；所有 position(name==owner) 改成索引直取。TradeContext 不必 clone 一份 operators 去 collect，借用即可（把排序后的 Vec<(&Atom, usize)> 与可变 operators 分离生命周期）。
把 layout 在搜索内借用而非每组合 clone：TradeContext 持 &TradeLayoutContext + 一个小的可变 state_pool（only when StateWrite 实际发生；多数组合 state_pool 为空，可 lazy）。global_inject.trade_eff_pct() 是只读标量，根本不需要 clone 整个 layout。
P1 — buff_id 字符串 → 预解析索引（你这种 FFM/二进制背景会喜欢） 建池时用 SkillTable::resolve_indices 把 Vec<String> buff_ids 变成 Vec<u32>（skill_table 行号），运行时 table.skills()[i] 直取，省掉热路径上所有 HashMap<String,_>::get 和 buff_id 字符串比较（shortcut.rs 里大量 bid == "trade_ord_law[000]"/starts_with 同理可换成 enum 标志位，建池时算一次存进 TradePoolEntry，如 is_docus/is_witch_e2/tailor_tier/constant_flat_eff——互斥判定与 classify 直接读位域，瓶颈 #2/#5 一起解决）。operator 名也建议 intern 成 u32，used: HashSet<u32>。

P1 — 组合枚举零分配 combinations_indices 改成产出 [usize;3]（或按 combination rank 直接从 par_iter 的 0..C(n,3) 反解出三元组），避免 Vec<Vec<usize>> 物化；rayon 用 (0..count).into_par_iter() 更均衡、cache 更友好。

P1 — 互斥校验只算一次 搜索层 eval_combo_hit 已校验过，给 solve_trade_with_shift 加一个 skip_exclusivity: bool（或拆出 solve_trade_unchecked），resolve_trade_shortcut 入口那次双保险在搜索路径下也跳过。

P2 — meta/锚点搜索改为锚点驱动 建池时标记机制锚点干员集合；witch/closure/docus role 只枚举"含该锚点"的 C(n-1,2)（已有 must_include_name 通路），把三次全量 C(n,3) 降到三次 ~C(n,2)。

P2 — 预编译每干员 atom 切片 collect_atoms 的 table.get+收集对同一 buff 集合是不变的；建池时给每个 TradePoolEntry 预存 Vec<&'static EffectAtom>（或索引区间）。每组合只需 3 段切片归并 + 一次排序（排序可换成 3 路 merge，因每段已按 phase 有序即可省排序）。

落地顺序我建议 P0 两项先做（改动局部、风险低、verify --all 即可回归），单独就能把 bench/schedule rotation 从"秒级带 syscall 抖动"压到"毫秒级 CPU-bound"；之后 P1 的 index 化是把它推向"内层纯 SIMD-友好算术"的关键，届时百万级组合可进 10–50M solves/s 量级，宏观排班（assign_base_greedy 多设施并行）也不再需要任何算法层取巧。
