## 1. 一句话结论

莱茵生命不是只能以固定三人组出现的体系，而是由两个可独立启用、又能互相增益的子系统组成：

- **莱茵科技同房组合**：制造房可以只放一人、二人或三人；三级站也允许用莱茵双人加任意制造干员。多萝西与淬羽赫默均要求 E2，但缺少其中一人不关闭整个莱茵体系。
- **莱茵生命阵营计数**：娜斯提与缪尔赛思分别读取当前班次基建内的莱茵生命阵营人数。娜斯提满额需要包含自身在内 5 名；缪尔赛思满额需要自身之外 5 名，即包含缪尔赛思在内至少 6 名。

由于满额阵营计数需要占用较多非生产位置，带娜斯提、缪尔赛思的方案应在主要生产体系之后评估：先确认当前班次的宿舍、会客室、加工站和训练室等位置是否足够，再计算完整收益，不能先按满额收益抢占生产岗位。

## 2. 两类“莱茵”必须分开

### 2.1 莱茵生命阵营（`cc.g.rhine`）

影响娜斯提和缪尔赛思的全基建人数计数。当前数据中 tag 已存在，不是待新增状态。

| 干员 | 阵营归属 |
|------|----------|
| 赫默、伊芙利特、塞雷娅、白面鸮、梅尔、麦哲伦、多萝西、星源、缪尔赛思、娜斯提 | 莱茵生命阵营 |
| 淬羽赫默 | 非莱茵生命；持有莱茵科技·γ |
| 溯光星源 | 非莱茵生命；持有莱茵科技·β和勘探背包 |

莱茵生命阵营实际人数可以超过 5；“5”只是单个技能的计数上限，不是阵营总人数上限。

### 2.2 莱茵科技技能（α/β/γ）

影响多萝西和溯光星源的同制造房技能计数。

| 干员 | 生效练度与技能 | 配方作用域 |
|------|----------------|------------|
| 星源 | α 15% / β 25% | 通用制造 |
| 赫默 | α 15% / β 25% | 通用制造 |
| 娜斯提 | E2 β 25% + 造价高昂 | 仅娜斯提的阵营加成限定赤金 |
| 白面鸮 | α 15% / β 25% | 通用制造 |
| 多萝西 | E2 β 25% + 同房每个莱茵科技技能 5% | 通用制造 |
| 溯光星源 | E2 β 25% + 同房每个莱茵科技技能 5 容量 | 通用制造 |
| 淬羽赫默 | E2 γ 30% | 通用制造 |

## 3. 已确认的领域不变量

| 类别 | 确认口径 |
|------|----------|
| 核心练度 | 将多萝西或淬羽赫默作为莱茵科技主力时，两人都必须 E2 |
| 最低人数 | 体系允许娜斯提单走，也允许二人站和三级站双人工作，不要求凑齐三人后才启用 |
| 房间容量 | 二级两人站允许多萝西 + 淬羽赫默直接工作；三级站最多三人，不存在四人同制造房 |
| 第三人 | 先比较多萝西 + 娜斯提 + 淬羽赫默；不足或不占优时，可用星源、白面鸮、赫默、溯光星源或任意合法制造干员替换 |
| 动态顺序 | 第三人和整房方案按目标配方、最终完整效率和其他体系机会成本动态排序，不使用固定姓名顺序 |
| 配方 | 娜斯提只进入赤金站；其他莱茵科技成员可用于赤金、作战记录或源石 |
| 关闭与降级 | 缺多萝西或淬羽赫默不整体关闭。可降级为娜斯提单人、莱茵双人或莱茵双人 + 普通第三人，并与普通三人制造组比较 |
| 独立启用 | 娜斯提、缪尔赛思可以各自独立启用，不要求莱茵科技双核心同时存在 |
| 协同倾向 | 若当前班次已为娜斯提准备足够莱茵挂件，优先同时评估缪尔赛思，因为两者可以复用同一批阵营人数 |
| priority | 娜斯提单人候选为 13；莱茵二人或三人制造组为 12 |
| 阵营位置 | 宿舍、会客室、加工站、训练主位和训练副手中的莱茵生命干员都计数 |
| 班次 | 同一制造房的莱茵成员必须绑定同一轮换队；工作一班还是多班由各班最终效率排名决定，不固定为上二休一 |
| 共存 | 与自动化组没有关系，也不共享成员；是否启用只由房间、空位和最终效率竞争决定 |

## 4. 莱茵科技制造组合

### 4.1 不是固定三人门槛

莱茵制造应作为从一人到三人的动态候选族参与比较：

1. 娜斯提可以在阵营人数足够时单独进入赤金站，单人最高 40%。
2. 二人站允许多萝西 + 淬羽赫默，也允许娜斯提 + 多萝西或娜斯提 + 淬羽赫默。
3. 三级站首选比较多萝西 + 娜斯提 + 淬羽赫默；也可以是莱茵双人 + 任意普通制造角色。
4. 凑不齐理想第三人时不关闭，直接比较“莱茵双人 + 第三人”和其他完整三人组的最终效率。
5. “更多莱茵成员”指其他房间或非生产设施中的阵营挂件，不表示一个三级制造站可以放四人。

### 4.2 已确认数值锚点

以下均为制造技能效率，不含人头基础 100%及其他全局制造注入：

| 组合 | 正式锚点 | 公式 |
|------|----------|------|
| 娜斯提 E2 单人赤金 | 最高 40% | `25% + 3% × min(含自身的莱茵人数, 5)` |
| 多萝西 E2 + 淬羽赫默 E2 | 65% | 多萝西 `25+10=35%`，淬羽赫默 `30%` |
| 多萝西 + 淬羽赫默 + 普通 E2 β | 95% | 多萝西 `25+15=40%` + `30%` + `25%` |
| 多萝西 + 淬羽赫默 + 娜斯提 | 95%～110% | `95% + 3% × min(含娜斯提的莱茵人数, 5)` |
| 多萝西 + 淬羽赫默 + 溯光星源 | 95% + 15 容量 | 三个 β/γ 技能使多萝西获得 15%，溯光星源获得 15 容量 |

### 4.3 双人数值锚点

用户已确认“上限 40%”仅指娜斯提单人。双人组按各自实际公式计算：娜斯提 + 淬羽赫默最高 70%，娜斯提 + 多萝西最高 75%。

按已确认技能公式推导：

| 双人组合 | 推导公式 | 五名莱茵时 |
|----------|----------|------------|
| 娜斯提 + 淬羽赫默 | `25 + 30 + 3R = 55 + 3R` | **70%** |
| 娜斯提 + 多萝西 | `25 + (25+10) + 3R = 60 + 3R` | **75%** |

多萝西与娜斯提同房时有两个莱茵科技技能，因此多萝西贡献 `25+10=35%`；娜斯提满额贡献 40%，合计 75%。

## 5. 阵营计数与空位预检

### 5.1 两个接收者使用不同人数口径

设当前班次基建内共有 `R` 名不同的莱茵生命阵营干员：

| 接收者 | 计数 | 满额所需 | 满额收益 |
|--------|------|----------|----------|
| 娜斯提 | 包含娜斯提自身，`min(R, 5)` | 含娜斯提共 5 名 | 赤金额外 15% |
| 缪尔赛思 | 排除缪尔赛思自身，`min(R-1, 5)` | 缪尔赛思 + 另外 5 名，共 6 名 | 基础 10% + 额外 15% = 25% |

不能用一个“截断到 5 的全基建人数”同时结算两人。娜斯提需要总数上限 5；缪尔赛思需要保存至少 6 人的总数，或直接保存排除自身后的独立计数。

### 5.2 空位成本是启用条件的一部分

娜斯提、缪尔赛思的满额组合需要大量挂件，应按以下顺序评估：

```text
先安排更高优先级生产体系
  -> 统计当前班次仍可使用的宿舍/会客/加工/训练等位置
  -> 统计可以合法进驻这些位置的莱茵生命成员
  -> 分别计算娜斯提 5 人档、缪尔赛思 6 人档及不足人数的实际收益
  -> 将完整收益与占用空位的机会成本一起参与最终排序
```

不得只因账号拥有五名或六名莱茵干员，就假定当前班次能够同时把他们全部放入基建。

### 5.3 班次一致性

阵营人数是当前班次状态，不是账号拥有量。用于娜斯提或缪尔赛思计数的挂件必须在接收者工作的同一班次实际进入基建。制造房内成员必须同队；挂件如何分配休息班次可遵循正常换班逻辑，但不能在接收者工作时缺席后仍按满额计算。

### 7.2 未实现或错误实现

| 生命周期 | 当前缺口 |
|----------|----------|
| select / plan | 没有莱茵候选族、空位预检、priority 或降级计划 |
| fill | 普通制造候选目录只有淬羽赫默；多萝西、娜斯提不会因“自然 C(n,3)”获得结构性入编 |
| fill | 缪尔赛思不在普通发电候选目录；娜斯提、缪尔赛思及阵营挂件都没有独立入编责任 |
| resolve | 全莱茵人数先截断到 5，缪尔赛思再减自身，导致满配只能得到 4 人、22% |
| resolve / search | 发电与制造候选按落位前快照评分，可能漏算候选自身和同批入编成员 |
| rotation | 没有莱茵制造同房绑定，也没有保证挂件与接收者在同一工作班次出现 |
| export | 只能忠实导出已经生成的普通房间，不能恢复上游未表达的莱茵组合和人数语义 |

因此，现阶段只能说单个技能解释器已经建模，不能声称莱茵体系已经完整实现。

## 8. 建议的单一责任边界（仅文档建议）

未来若另开代码任务，建议由一个莱茵候选计划统一负责：

1. 根据制造房等级、配方和实际练度生成一人、二人、三人候选，不把多萝西 + 淬羽赫默误设为缺一即关闭的硬核心。
2. 将娜斯提候选限制为赤金；其余莱茵科技成员保持通用制造。
3. 在主要体系落位后读取真实剩余设施容量，为娜斯提和缪尔赛思分别构造 5 人与 6 人档。
4. 使用最终候选编制重新计算阵营人数，避免落位前快照漏算接收者或同批成员。
5. 仅当完整候选胜过普通制造组时提交房间；失败时自然降级，不保留半套强制占位。
6. 将最终同制造房成员写入轮换绑定；阵营挂件必须与接收者的工作班次对齐。

不得通过固定房号、按姓名散落特判、把 tag 当作 required anchor，或假定账号拥有量等于本班进驻量来实现。
