本 ADR 记录排班改版的核心边界调整：废弃“先生成统一排班结构，再按执行工具转换”的旧口径，改为按实际换班逻辑分层。

## 背景

旧版排班改版设计把主力、最小换班组、替补、宿舍、菲亚梅塔视作一套统一排班核心。Mower、MAA 与人工换班的差异被放在最后的输出转换层：Mower 接收动态触发结构，MAA 接收循环时间表，人工用户接收可读时间表。

后续评审确认，这个抽象会把三类本质不同的使用逻辑混在一个求解器里：

- 游戏内置自动轮换按“组合心情耗尽”触发，只替换耗尽组合。
- MAA 与人工手动换班按固定时间或相对间隔执行，真实存在二班、三班、四班等完整班次。
- Mower 动态换班依赖实时心情监控，不存在固定班次，由 Mower 决定何时换班、谁进宿舍、谁接替。

这些逻辑共享干员池、机制结算、组合合法性、心情消耗估算等基础能力，但不共享换班触发条件、替补关系、宿舍规则、菲亚梅塔目标、班次概念和输出责任。

## 决策

排班相关能力拆成三个显式模式：

```text
auto_rotation / timed_rotation / mower_dynamic
```

用户未声明换班逻辑时，默认进入 `timed_rotation`。

### A. 共用能力层

三个模式只共享基础能力：

- `OperBox`、干员拥有情况与练度；
- 技能、机制、shortcut / L2 / L3 结算；
- 单房间效率、组合合法性、体系优先级；
- 理论心情消耗与最长可用时间估算；
- 特殊高消耗角色提示。

共用能力不决定具体换班方式，也不输出跨模式统一排班结构。

### B. 定时换班为当前优先实现

`timed_rotation` 面向 MAA 与人工手动换班。二者共用同一套求解规则，核心输入是：

```text
班次数量 + 操作间隔 / 相对间隔 + 是否有固定作息限制
```

定时换班的生成顺序为：

```text
主力编制 -> 最小换班组 -> 菲亚梅塔 -> 排班方案 -> 宿舍与宿管
```

定时换班输出固定班次表，而不是动态替补关系。输出应能表达：

- 每个班次的完整房间安排；
- 固定执行时间或相对间隔；
- MAA 可用计划；
- 人工可读时间表；
- 菲亚梅塔动作；
- 无人机动作；
- 简化宿舍安排；
- 放弃的高消耗角色或组合及原因。

定时换班默认峰值优先。若某个主力组合无法适配用户声明的班次时长，应提示放弃该组合或调整班次，不把问题转入 Mower 动态逻辑。

### C. 自动轮换单独建模

`auto_rotation` 只服务游戏内置自动轮换，不作为 MAA 或 Mower 输出。

自动轮换后续可作为具备特殊要求的二班制落地，但不沿用 `timed_rotation` 的完整结构。它应输出主力组合、独立替补组合、自动轮换可行性，以及不适合自动轮换的跨设施组合提示。

自动轮换下默认不主动规划菲亚梅塔；跨三个设施及以上的体系默认禁用，跨站范围原则上不超过两类不同设施。

### D. Mower 动态换班暂缓

`mower_dynamic` 不套用 `timed_rotation` 求解器，也不使用二班、三班、四班作为核心模型。

Mower 模式后续单独设计，至少需要重新确认：

- 主力组合与替补组合声明；
- 多个主力组合共用一个替补组合的上限；
- 替补工作时长如何覆盖主力休息时长总和；
- 菲亚梅塔目标范围；
- 每个宿舍一名单回 + 一名群回的宿管优先级；
- 宿舍黑名单、回满、耗尽、红脸 0 心情上班等规则。

Mower 下菲亚梅塔不套用定时换班的但书、巫恋 + 龙舌兰、清流、可露希尔等目标规则。

## 后果

这个决策让排班实现从“一个大求解器 + 多个输出转换器”改为“共用基础能力 + 模式专属求解器”。

直接后果：

- MAA 与人工手动换班可以先稳定落地，不被 Mower 的实时心情、宿舍黑名单、红脸规则拖入复杂状态机。
- `timed_rotation` 的输出形状更清晰：完整班次、执行时间或相对间隔、菲亚梅塔、无人机、简化宿舍。
- 高消耗或心情异常组合在定时换班中优先拒绝或降级说明，而不是尝试用动态恢复规则补救。
- 未来 API / CLI / UI 需要能显式区分 `auto_rotation / timed_rotation / mower_dynamic`。
- 旧版设计中“统一排班逻辑服务 Mower 与 MAA，只在执行转换层分叉”的口径不再作为实现依据。

代价是：Mower 与自动轮换不能复用定时换班求解器，只能复用基础能力；后续需要为两者分别补充独立规格与验收案例。

## 被取代的方案

### 统一排班核心

旧方案希望先得到统一结构：

```text
主力岗位 + 最小联动分组 + 替补顺序 + 宿舍策略 + 特殊约束
```

然后按工具能力转换成 Mower 动态换班或 MAA 循环计划。

该方案被取代，因为 Mower 与定时换班不只是输出格式不同，而是触发条件、替补关系、宿舍责任、菲亚目标和班次概念都不同。

### 在定时换班中吸收 Mower 规则

不在 `timed_rotation` 中引入回满、耗尽、红脸 0 心情上班、宿舍黑名单、实时心情监控或动态替补选择。这些规则留给 `mower_dynamic`。

### 自动轮换作为 MAA / Mower 输出

自动轮换只面向游戏内置自动轮换功能。它不是 MAA 时间表，也不是 Mower 动态换班声明。

## 验收口径

后续实现排班改版时，满足以下条件才算符合本 ADR：

- 未声明模式时进入 `timed_rotation`。
- MAA 与人工手动换班共用定时换班规则，不拆成两套求解器。
- 定时换班能输出完整班次表、执行时间或相对间隔、菲亚梅塔动作、无人机动作、简化宿舍安排和高消耗放弃说明。
- MAA 定时换班最多建议 4 个班次状态；超过时提醒或警告。
- 非 24h 周期但规律轮换时，用相对间隔表达，不强行转为每日固定时间点。
- 定时换班理论换班时间带经验安全余量，优先支持约半小时容错。
- Mower 动态换班不复用定时换班求解器。
- 自动轮换不作为 MAA 或 Mower 输出。
