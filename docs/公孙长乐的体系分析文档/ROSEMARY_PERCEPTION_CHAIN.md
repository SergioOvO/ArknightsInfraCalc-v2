## 1. 一句话结论

迷迭香感知链是一套**跨设施全局资源协作体系**：迷迭香（制造）+ 黑键（贸易）绑定同上同下，以感知信息（Perception）为核心中间件，搭配感知 producer（夕/絮雨/爱丽丝/车尔尼）堆高全局感知，经思维链环/无声共鸣转化为制造效率和贸易效率。在 243/153/333 三电布局下，满配约 **80–90% 制造效率 / 40–45% 贸易效率**，日均增量约 **3000 赤金或 2400 经验**（相对无感知链）。

维护期已经确认“迷迭香 E2 + 黑键 E2 缺一关闭、peak 同时进编、同队上 2 休 1、黑键位于非巫恋贸易站”等硬约束，但文档中的其他 producer、办公室、塑心、优先级、数值锚点和降级描述仍需作为一个完整体系重新核对。红松林改动触及共享的 `AssignmentPlan`、producer fill、shift bind 和 team rotation；现有迷迭香回归通过只能说明已覆盖约束未被破坏，不能代替该体系自己的完整审计。

## 2. 何时值得开（决策表：布局 × 盒内干员 × 已有 meta）

### 2.1 硬性前提（全部满足才应考虑）

| 条件 | 说明 |
|------|------|
| 布局 | 三发电（243 / 153 / 333）。二发电（252 / 342）因宿舍等级上限降低，爱丽丝/车尔尼产出缩水至每人 1–2%，但仍可用 |

- **缺迷迭香或黑键**（任意一个都没有）→ 不开
- 有迷迭香+黑键但**无絮雨 AND 无八幡海铃 AND 无焰狐龙梓兰** → 感知源太少，不建议开
- 布局为四发电或零发电的非标布局 → 未验证，不保证

### 2.3 与已有 meta 的共存

| 体系 | 关系 |
|------|------|

| 可露希尔组 | **完全兼容**。黑键 + 可露希尔次优搭法 |
| 巫恋组 | **不与黑键同站**。巫恋不宜与黑键放同一贸易站 |
| 叙拉古中枢（八幡海铃） | **完全兼容**。夕中枢 + 八幡海铃同中枢不影响；但无絮雨+琴柳时感知少 20 |

| 喀兰中枢（灵知）| 互斥于中枢位，由 `exclusive_group: meta_chain` 控制 |
| 人间烟火链（乌有+令+重岳） | 第二班替代场景：夕心情<12 后转产人间烟火，搭配乌有/令/重岳提高贸易效率 |

## 4. 缺人降级路径（逐档：全链 → 子链 → 关闭）

降级按**从外围向核心**的顺序，从最不关键的 producer 开始砍：

```
满配全链（迷迭香 90% / 黑键 45%）
   │ 砍中枢令（控制在>12 心情）
   ├─ 第一档  （迷迭香 80% / 黑键 40%）
   │ 砍宿舍三人组（爱丽丝+车尔尼+塑心）
   ├─ 第二档  （迷迭香 75% / 黑键 37.5%）
   │ 砍中枢夕
   ├─ 第三档  （迷迭香 ~55% / 黑键 ~27.5%）
   │
   └─ 仅保留三核心：迷迭香 E2 + 黑键 E2 + 絮雨 E2
      若再缺迷迭香或黑键 → 整链关闭
```

### 各档详细说明

| 降级档 | 保留 | 砍掉 | 典型效率 |
|--------|------|------|---------|
| 满配 | 核心 + 中枢令(≤12) + 夕 + 爱丽丝+车尔尼+塑心 + 絮雨+琴柳 | — | 迷迭香 90%，黑键 45% |
| 档 1 | 核心 + 夕 + 爱丽丝+车尔尼+塑心 + 絮雨 | 令（不控心情） | 迷迭香 80%，黑键 40% |
| 档 2 | 核心 + 夕 + 絮雨 | 爱丽丝+车尔尼+塑心 | 迷迭香 75%，黑键 37.5% |
| 档 3 | 核心（迷迭香+黑键+絮雨） | 夕 | ~55% / ~27.5% |
| 关闭 | — | 缺迷迭香或黑键 | 回退普通贪心 |

### 代码路线：优先执行高于但书组还是低于？

- **全链 / 档 1 / 档 2**：应保持 priority 21（高于但书组 20），因为这代表感知链高收益状态。
- **档 3**：感知收益下降至 ~55%/27.5%，priority 应降至但书组之下（如 15–17），让但书优先认领贸易站。
- **关闭**：不认领。

## 8. 明确不选清单

以下情况**不应自动启动**迷迭香感知链成套：

1. **缺迷迭香 E2** 或 **缺黑键 E2** → 不开
2. **无絮雨 AND 无八幡海铃 AND 无焰狐龙梓兰** → 感知源不足，不开
3. **四发电布局** → 未经验证，不保证
4. **巫恋已占黑键候选贸易站且无其他空贸站** → 黑键不与巫恋同站
5. **恢复班**（`shift_mode == Recovery`）→ 不认领
6. **但书/可露希尔所需散件与黑键冲突** → 优先给但书分配最优散件，黑键拿次优；若无可接受散件则黑键不占贸位

## 9. 建议落地的 base_systems / assign 改法（条目化，不写大段代码）

### 9.1 `base_systems.json` 改法

1. **`rosemary_perception` 改为半固定**：
   - 仍保留为 `base_systems` 条目，但 slot 缩减为：
     - `control`：夕（elite: **0**）（可选 slot，可跳过）
     - `office`：絮雨 E2（可选 slot，仅在蓝图有办公室时加入）
     - `trade_post`：黑键（不指定 room_id，不指定队友）
     - `factory`：迷迭香（不指定 room_id，不指定队友）
   - `dormitory` slot **取消固定**（爱丽丝/车尔尼作为效率散件，不钉死成套）
   - 增加 `optional_slots` 或每个 slot 的 `required: false` 标记

2. **新增降级 variant**：
   - `rosemary_perception_tier1`（priority 21）：含夕 + 爱丽丝车尔尼 + 絮雨/琴柳
   - `rosemary_perception_tier2`（priority 21）：含夕 + 絮雨
   - `rosemary_perception_tier3`（priority 15）：仅迷迭香+黑键+絮雨（无额外感知 producer）
   - 或将降级逻辑移至 `system.rs` 内的 `system_feasible` 软化

3. **office slot 动态追加**：
   - `SystemSlotDef` 增加 `facility_required: Option<String>`，当蓝图包含该设施时才认领
   - 或由 `claim_base_systems` 检测蓝图是否有 office，动态追加絮雨 slot

### 9.2 `assign.rs` 改法

4. **`claim_base_systems` 软化为非全有全无**：
   - 当前 `system_feasible` 返回 `bool`，跳过整个 system。改为返回 `Option<BaseSystemDef>`，裁剪不可用的 slot 后返回一个精简版 system。
   - 若裁剪后核心 slot（迷迭香或黑键）缺失 → 跳过。

5. **trade_meta 落位增加「黑键非巫恋」约束**：
   - `assign_trade_meta` 中，若黑键已认领但尚未落位贸易站，分配时加入 filter：`黑键不在巫恋站`。
   - 或由 `claim_base_systems` 在认领时直接钉入一个贸易站（优先但书站，次优可露希尔站）。

6. **恢复班**保持当前跳过逻辑，不认领。

### 9.3 `resolve.rs` 改法

7. **办公室感知注入**：
   - `apply_perception_producers` 补充 `FacilityKind::Office` 分支：絮雨（记忆碎片 → 感知，每级办公室 +? 需查 skill_table）。
   - 需要 `skill_table.json` 中絮雨的 buff 已建模 → 属于数据层任务。

### 9.4 基础设施变更

8. **`SystemSlotDef` 加字段**：
   - `optional: bool`（可选 slot，缺人时不导致整链跳过）
   - `facility_optional: Option<String>`（如 office，仅在蓝图有该设施时激活）

9. **`exclusive_group` 复用**：
   - 保持 `meta_chain` 互斥组不变（叙拉古/喀兰/推王/怪猎四选一）。
   - 迷迭香链自己不入组（不与任何中枢体系互斥）。
