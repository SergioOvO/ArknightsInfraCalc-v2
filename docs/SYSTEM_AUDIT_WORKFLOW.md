## 2. 不可跳过的阶段

```text
选择一个体系
  -> 阅读体系 Markdown 和定向机制登记
  -> 沿完整生命周期只读审计
  -> 向用户提交详细审计报告
  -> 等待用户逐项裁决
  -> 先同步被纠正的 Markdown
  -> 向用户提交详细修改计划
  -> 等待用户明确批准
  -> 调用 subagent 实施
  -> 主 Agent 审查、补漏、验证、提交
  -> 提交强制证明表
  -> 再进入下一个体系
```

两个等待点是硬门禁：

1. 审计报告后不得直接拟定最终语义或修改代码；先让用户裁决文档错误、规则歧义和数值口径。
2. 修改计划后不得直接编辑；只有用户明确同意，主 Agent 才调用 subagent 修改。

## 3. 第一阶段：只读审计

### 3.1 阅读范围

必须读取：

- 当前体系对应的 Markdown，完整读完。
- `docs/MAINTENANCE_MODE.md`、`docs/PROJECT_MAP.md` 中与该体系有关的生命周期和模块边界。
- `data/MECHANICS_REGISTRY.csv` 中相关干员或技能行。
- 体系涉及的 `skill_table.json`、`operator_instances.json`、tag、shortcut、`base_systems.json` 条目。
- 从体系选择到用户输出的实际代码路径。

不要全量通读无关理论文档，也不要把现有测试通过当成语义正确的证据。

### 3.2 先提取领域不变量

审计报告必须先给出下表，不得直接从代码现状开始讲：

| 类别 | 必须回答的问题 |
|------|----------------|
| 硬核心 | 哪些干员、精英阶段、设施或配方缺一不可？ |
| 最低数量 | 候选组至少需要几名？拥有更多成员时是否必须全员纳入？ |
| 可选 producer | 哪些人只提供资源或效率，不应被强制锚定？ |
| 位置作用域 | 要求同房、跨站、只需在基建内，还是只需进入某种配方？ |
| 互斥与共存 | 与哪些体系争用成员、房间、配方或 meta group？ |
| 班次 | 谁必须同队、是否上二休一、是否允许自然轮换？ |
| 关闭与降级 | 缺谁必须整体关闭；哪些降级档真实存在？ |
| 数值锚点 | 满配和降级的公式如何逐项相加，作用对象和作用域是什么？ |

Markdown 内部或多个 Markdown 之间若冲突，逐条引用冲突位置并停下来请用户裁决。代码、CSV、测试和运行输出都不能替 Markdown 作决定。

### 3.3 沿完整生命周期找违规位置

每条不变量都要沿以下路径检查，不能只看技能结算或最终输出：

```text
select -> plan -> execute -> fill -> resolve -> rotation -> export
```

报告必须指出具体文件、类型或函数，并区分：

- 体系是否被正确激活或关闭。
- 硬核心是否作为 required anchor 真正进入编制。
- role filter 是否只允许合法候选。
- fill 是否保留全部 anchor，尤其是同房多 anchor 和跨站分布。
- resolve 中的技能数值、作用域、配方和 tag 是否正确。
- `shift_bind` 是否只负责绑定已入编成员，而没有被误用为进编保证。
- rotation 是否实现同队上二休一，而不是某个样例碰巧出现该结果。
- export 是否忠实输出 core 排班，没有丢人、换房或改班次语义。

### 3.4 专门检查“结果正确但只是巧合”

出现以下情况，必须报告为未完整实现，即使标准 243 输出看起来正确：

- 干员没有 required anchor，只因当前效率排序是 top hit 而自然上岗。
- 同队关系没有 bind，只因 `used`、房间遍历顺序或队伍切分顺序碰巧一致。
- 配方没有约束，只因标准 fixture 恰好只有目标配方空位。
- 跨站规则靠固定 `room_id`、房间下标或当前布局房间数量实现。
- 体系激活只检查部分硬核心，缺人时仍能生成看似合理的散件配置。
- shortcut 结算正确，但体系选择和进编没有保证组合实际存在。
- 一个 anchor 被后续 fill 覆盖，但最终又被搜索重新选中。
- 测试只断言最终总效率或 top hit，没有断言成员、配方、作用域、关闭条件和轮换。
- 测试复用了与实现相同的错误常量或错误公式，形成自证循环。

识别方法是做最小反例，而不是只重复标准全精二 fixture：删掉一个硬核心、只保留最低人数、增加竞争候选、改变房间顺序、提供两个目标配方房间、强制跨站容量，并观察不变量是否仍由结构保证。

## 4. 审计报告格式

每个体系单独提交一份详细报告，按严重程度列问题：

| 编号 | 文档不变量 | 当前实现 | 违规生命周期 | 为什么不是完整实现 | 证据 |
|------|------------|----------|--------------|----------------------|------|

报告末尾必须包含：

1. **已严格实现**：列出代码中确有唯一责任边界、且已有反例回归的规则。
2. **未实现或错误实现**：包括补丁式逻辑、冲突旧路径、作用域错误和缺少关闭条件。
3. **疑似巧合**：当前结果正确，但没有结构保证的规则。
4. **文档疑点**：只列需要用户裁决的问题，不自行修正文档。
5. **待确认问题**：问题要具体到可直接回答“是/否、人数、绑定范围、是否降级、精确公式”。

此阶段不写代码，不调用 subagent，不把审计发现直接变成实现。

## 5. 用户裁决后的修改计划

收到用户回答后，先把裁决同步到对应体系 Markdown，消除旧口径。然后提交修改计划，必须包含维护门禁的四项：

### 5.1 领域不变量

把用户确认后的硬核心、最低人数、全员纳入规则、可选 producer、位置作用域、互斥、班次、关闭和降级条件逐条写清。

### 5.2 违规位置

对每条待修规则指出当前在哪个生命周期丢失，给出文件、类型、函数和非法状态如何产生。

### 5.3 单一责任边界

逐条说明修复后由哪个 System、`AssignmentPlan` 字段、anchor、role filter、通用 fill 或 solver 边界唯一保证。若同一语义需要在多个阶段写体系名特判，计划不合格，应重新设计责任边界。

### 5.4 删除清单

明确列出要删除或改写的旧 system 条目、fallback、按中文名分支、固定房号、错误 tag/shortcut、错误注释和错误测试。不能只叠加新路径并保留冲突实现。

计划还必须列出：

- 修改文件范围和每个文件的职责。
- 单元、生命周期集成、端到端 CLI 三层回归矩阵。
- 最小人数、缺核心关闭、允许跨站或禁止同房、轮换绑定等反例。
- 与其他体系共享边界的回归，防止修好当前体系却破坏另一个体系。
- 预计不处理的范围和已知风险。

提交计划后等待用户明确批准。

## 6. 批准后的 subagent 实施协议

用户批准后，主 Agent 才能调用 subagent。推荐拆分：

| 角色 | 任务 |
|------|------|
| 实现 subagent | 按批准计划修改单一事实源、删除冲突路径、补核心测试 |
| 端到端 subagent | 设计最小 operbox/layout 反例并跑 `plan` 或 `layout team-rotation` |
| 主 Agent | 审查所有 diff，检查是否出现样例特判，补遗漏机制和共享回归，统一验证与提交 |

subagent 的任务描述必须附上用户确认的不变量、禁止事项、文件边界和验收断言。subagent 的“完成”不是最终结论；主 Agent 必须独立阅读 diff 和测试，尤其检查：

- 是否写入干员名、room id、班次下标、fixture 特征或当前 top hit 特判。
- 是否用 bind/tag/priority 代替 required anchor。
- 是否只新增路径而未删除冲突旧逻辑。
- 是否只验证最终效率，没有验证激活、关闭、进编、位置作用域和轮换。
- 是否影响共享的 fill、rotation、global inject 或其他体系。

## 7. 验证与完成证明

最低验证层级：

1. `cargo fmt --all`。
2. `cargo test -p infra-core --no-run`。
3. 当前体系定向测试，单线程运行以排除共享 fixture 污染。
4. 受影响共享边界的回归测试。
5. 用户真实入口：排班体系至少运行一次 `plan` 或 `layout team-rotation`。
6. `git diff --check`。
7. 全量测试；失败要区分本轮失败、既有失败和未验证风险。

端到端断言应读取设施类型、配方、成员和工作/休息状态，不固定 room id、队伍名称或班次编号。

完成回复必须逐条给出：

| 不变量 | 代码保证 | 删除的冲突 | 回归 | 端到端结果 |
|--------|----------|------------|------|------------|

并单独说明：根因层、旧模型为何允许非法状态、新的单一事实源、所有未通过测试、实际运行过的用户入口和 commit hash。缺少任何一项，不得声称“完全实现”。

## 8. 红松林案例留下的通用教训

红松林标准全精二结果最初看似正确，但补最小人数和跨站端到端测试后发现：同一制造房已有多个 required anchor 时，fill 只保留首个成员并覆盖整房。第二名成员有时会被后续搜索重新选中，因此普通输出可能掩盖结构错误。

最终修复没有为红松干员或固定房间写特判，而是让制造 fill 通用地保留全部 recipe anchor；体系本身由 `PinusPlan` 统一产出激活条件、双中枢 anchor、实际拥有的 2/3 名制造 anchor 和全员 `shift_bind`。这个案例说明，后续体系必须用最低人数、竞争候选和跨站容量反例证明，而不能只看满配总效率。
