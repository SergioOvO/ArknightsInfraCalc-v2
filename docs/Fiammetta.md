# 菲亚梅塔换心情规则

> 当前状态：MAA 单目标线性优先级已接入；布局动态排序与完整心情排班尚未实现。
> 规则来源：公孙长乐 2026-07-11 评审稿。

## 当前已经实现

`plan`、`layout team-rotation`、legacy `layout rotation` 和前端 `serve plan` 写出 MAA JSON 时，会为每个 plan 选择一个菲亚梅塔目标：

```text
但书 > 巫恋 > 龙舌兰 > 清流 > 可露希尔
```

选择规则：

1. 按上述顺序检查当前班次的 `BaseAssignment`。
2. 第一个正在当班的候选成为 `Fiammetta.target`。
3. 找到目标时输出 `enable: true`、`order: pre`。
4. 当前班次没有候选时输出 `enable: false` 和空目标。
5. 直接使用 core API 时，空优先级清单保持关闭，调用方可自行决定是否启用。

示例：当前班有龙舌兰、但书和可露希尔时，目标为但书；只有清流和可露希尔时，目标为清流。

MAA 负责执行具体的宿舍拖动、心情交换和菲亚梅塔返回宿舍。InfraCalc 当前只声明是否启用、目标和使用顺序，不强制宿舍槽位。

当前选择是逐 plan 独立完成的。例如 `12h + 6h + 6h` 三班可能连续三个 plan 都输出启用，但这不代表菲亚梅塔在每个时间点都已经回满。前端和用户不能把当前字段视为可用性证明或永续承诺。

## 已确认的机制与策略原则

菲亚梅塔是独立的心情交换事件，不是普通宿管，也不是一支完整替补班。她的用途是让高价值主力继续工作，减少低效率替补的上班时间。

使用前提和时机：

- 菲亚梅塔本人必须满心情；
- 默认在换班前使用，即 `order = pre`；
- 使用后菲亚梅塔回宿舍恢复；
- 菲亚梅塔自身按每小时 2 点恢复，不叠加其他宿管恢复效果；
- 下一次理论就绪时间为 `(24 - 交换后菲亚梅塔心情) / 2` 小时。

后续排班层选择目标时，基础收益率建议为：

```text
菲亚收益率 = (主力效率 - 替补效率) / 主力每小时心情消耗
```

这表示主力与替补的效率差越大，菲亚梅塔的价值越高；效率差相同时，心情消耗较低、一次交换能维持更久的目标优先。

## 尚未实现的动态规则

公孙长乐给出的完整业务优先级依赖布局和成组关系：

```text
但书（二发电布局中的一级或二级贸易站）
> 巫恋 + 龙舌兰（三级贸易站）
> 但书（三级贸易站）
> 清流（三贸易站布局）
> 可露希尔
```

当前 MAA 导出层没有布局语义、替补效率、心情状态和多次菲亚使用序列，因此只采用稳定的单目标线性顺序。以下内容必须在后续排班策略层实现，不能塞进 `export/maa.rs`：

- 根据布局和房间等级动态调整但书、清流的优先级；
- 将巫恋与龙舌兰作为跨班成组服务目标；
- 根据主力与替补的效率差实时计算收益率；
- 判断菲亚梅塔是否满心情、目标剩余心情和下一次就绪时间；
- 根据班次间隔、中枢心情减免和设施心情消耗决定可服务人数；
- 生成 Mower 的动态主力/替补策略，或展开成 MAA 的多个固定计划状态。

通常不建议把红松林、迷迭香等多人组合或高心情消耗的歌蕾蒂娅作为常规单点目标，但这仍属于后续排班策略，不是当前导出过滤规则。

## 代码边界

| 层级 | 当前职责 |
|------|----------|
| `schedule/` | 生成班次与当班 assignment；未来承载动态菲亚策略 |
| `export/maa.rs` | 把已确定的单目标写成 `enable`、`target`、`order` |
| `infra-cli` | 为常规 MAA 导出启用已确认的线性优先级 |
| MAA / Mower | 执行宿舍操作与实际换班 |

本功能不等于完整宿舍恢复模拟、全基建连续心情最优化或菲亚梅塔永续证明。
