## 1. 这个程序在做什么

本项目不是单纯算一个贸易站三人组效率，而是做一套完整流程：

1. 读玩家练度表，知道玩家拥有哪些干员、精英化到什么程度。
2. 读基建布局，知道有几间贸易站、制造站、发电站、中枢、宿舍、办公室。
3. 读技能表，知道每个干员的基建技能原文机制。
4. 贸易等领域可按既有策略缩池；排班制造使用全部合法普通制造候选，不由工具人表裁剪。
5. 对每个核心体系、每个房间生成可行候选。
6. 处理同一干员不能同时上两个岗位的冲突。
7. 输出单班排班或 alpha/beta/gamma 三队轮换。

理想状态下，只要我们给足策略，程序应该自然排出接近最优的方案，而不是每遇到一个账号就手写一次特例。

## 2. 数据从哪里来

项目里有几类数据，它们的地位不同。

| 数据 | 作用 | 对应基建概念 |
|------|------|--------------|
| `MECHANICS_REGISTRY.csv` | 全干员基建技能原文真源 | PRTS/游戏技能表 |
| `operator_instances.json` | 某干员在精0/精1/精2时有哪些技能 | 干员练度和技能解锁 |
| `skill_table.json` | 把技能原文翻译成程序可计算的机制 atom | 技能机制建模 |
| `trade_shortcuts.json` | 难以完全拆公式的贸易组合锚点 | 公孙工具人表里的等效效率 |
| `trade_segments.json` | 跨站 producer 到贸易 consumer 的链段 | 叙拉古、推王等跨站条件 |
| `base_systems.json` | 目前登记的成套体系/组合 | 跨站体系、产线体系、同站组合 |
| `standalone_roster.json` | 人工维护的散件工具人白名单 | 高效散件池，用来缩小搜索范围 |

其中最重要的一点是：

`MECHANICS_REGISTRY.csv` 是原文真源；其他 JSON 都是建模、锚点或排班策略，不能反过来覆盖技能原文。

## 3. 程序的核心思路

基建排班可以理解成三层：

### 3.1 先算机制

这一层回答：

“如果某三个人在同一个贸易站/制造站，他们的效率是多少？”

例如：

- 巫恋会清零其他人的订单效率，再按人数给自己加效率。
- 龙舌兰在大额赤金订单时增加龙门币收益。
- 可露希尔会生成特别订单。
- 但书会把小额赤金订单变成违约订单，并提高交付数。
- 清流按贸易站数量给赤金制造加成。
- 温蒂/森蚺自动化按发电站数量给制造加成。

这层不决定谁优先上岗，只负责把机制算对。

### 3.2 再生成候选

这一层回答：

“有哪些值得考虑的站？”

候选来源有几种：

- 需要 producer 与 consumer 联合比较的跨站候选，比如叙拉古动态注入。
- 核心优先体系，比如但书、可露希尔、巫恋。
- 同站 bond，比如德克萨斯+拉普兰德、能天使+蕾缪安。
- 产线体系，比如自动化金线、红松林经验线。
- 散件工具人，比如普通高效贸易/制造/发电干员。

这里最容易出错：不能把所有东西都写成“固定三人组”。

比如：

- 贸易 core 只看蓝图实际贸易站数量与实际订单：恰有 1 间贸易站且为龙门币订单时可露希尔是 required core；至少 2 间贸易站且任一为龙门币订单时但书是 required core；全部为源石订单时两者都不上。required core 解析到实际龙门币站后，再在该房按最终效率选择最高效率队友。八幡海铃、伺夜、贝洛内都不是固定体系成员：程序同时考察普通中枢与包含动态贸易 producer 的中枢候选；伺夜、贝洛内进入正常贸易散件候选池，搜索可自然选中 0/1/2 人。两人不要求同站，是否与但书同房也由最终效率决定。
- 可露希尔不是固定等于“可露希尔+黑键+吉星”；单贸龙门币布局中她作为 required core 进入搜索，黑键只是高价值队友之一。
- 巫恋/龙舌兰不是只认一个固定 beta 队；beta 不够时应有 alpha/普通工具人 fallback。

### 3.3 最后选排班

这一层回答：

“在同一人不能重复上岗的前提下，哪些候选组合应该进入最终排班？”

当前项目不是做无限规模的全局最优，因为那会爆炸。实际做法是：

1. 贸易等非制造领域可按各自人工工具人表或 role policy 缩池。
2. 普通排班制造使用 `full_pool=true` 的全部合法普通候选；逐房过滤 `used`、基于当前上下文搜索并立即提交。
3. 每个贸易核心或其他适用房间只保留前若干个好候选，再按命名策略处理冲突。
4. 剩余空房按各领域自己的候选边界搜索补齐。

贸易等缩池领域的局部搜索通常较快；普通排班制造不以工具人表换取该速度，其完整候选搜索耗时以当前用户 `plan` 入口实测为准。两者都应少写硬编码，让领域边界内的搜索自然选队友。

## 4. 为什么工具人表对适用的非制造领域很重要

贸易站等适用领域的三人组合数会随候选池快速膨胀，因此仍可使用领域自己的工具人表缩池。本节不适用于普通排班制造：制造使用 `full_pool=true` 的全部合法普通候选。

人工维护的工具人表相当于告诉程序：

“这些人是值得搜索的高价值散件；其他人除非体系需要，否则不用每次都参与穷举。”

这样程序就能在很小的池子里搜索：

- 但书 + 最优两个队友。
- 可露希尔 + 最优两个队友。
- 巫恋 + 龙舌兰 + 最合适第三人。
- 普通贸易站三散件。

这比直接写死固定组合更健康，因为它能适应不同账号缺干员、练度不齐、布局不同的情况。

## 5. 四类体系应该怎么分

### 5.1 中间产物 / 全局资源

这类不是组合，而是全基建共享资源。

例子：

- 感知信息
- 人间烟火
- 木天蓼
- 魔物料理
- 无声共鸣
- 思维链环
- 虚拟发电站

正确处理方式：

先让 producer 上岗，程序结算全局资源，再让消费者读取这些资源。

例如黑键和迷迭香都读感知链收益，但这不是“黑键必须和某人固定一站”的问题。

### 5.2 跨站 meta

这类是 producer 在一个房间，consumer 在另一个房间。

例子：

- 八幡海铃在中枢，叙拉古贸易站吃加成。
- 戴菲恩在中枢，格拉斯哥贸易站吃加成。
- 灵知在中枢，喀兰贸易站订单上限变化。

正确处理方式：

producer 与 consumer 都不是硬核心时，分别完成“不含动态 producer”和“包含动态 producer”的自然贸易搜索，再按同一具名中枢 policy 比较；不能先固定 producer，也不能强塞 consumer。

### 5.3 同站组合

这类是真正要求同一个房间的组合。

例子：

- 德克萨斯 + 拉普兰德
- 能天使 + 蕾缪安
- 摩根 + 推进之王
- 阿兰娜 + 温米
- Miss.Christine + 酒神

正确处理方式：

固定核心关系，第三人让搜索补。

### 5.4 核心优先

这类不是固定组合，而是“这个核心很强，必须优先考虑”。

例子：

- 但书
- 可露希尔
- 巫恋/龙舌兰

正确处理方式：

强制核心进入候选，然后在缩小后的池子里找最佳队友。

## 6. 当前最需要改的地方

当前项目有一部分历史实现把“核心优先”误写成了“固定组合”。

主要问题：

- `witch_long_beta` 写得太像固定三人组。
- `blackkey_closure` 写得太像固定三人组。
- 企鹅/推王等较低优先组合有时会过早抢站。
- 但书同房 shortcut、叙拉古跨站标签收益和但书核心优先曾被错误合并为固定体系。
- 感知、木天蓼这类全局资源有时在文档语义上和组合混在一起。

这就是为什么小饼账号会出现“没有但书、没有可露、没有龙巫”的严重错误：程序在局部 fixed registry 里提前占掉了位置和人。

## 7. 期望中的健康运行流程

未来重构后，流程应变成这样：

```text
练度表 + 布局
  ↓
读技能真源和练度
  ↓
按领域构造候选池：
  - 普通排班制造：full_pool 全部合法普通候选
  - 贸易等其他领域：按各自工具人表 / role policy 缩池
  ↓
结算已上岗 producer 能带来的全局资源
  ↓
按策略生成候选：
  - 但书 role
  - 可露 role
  - 巫恋 role
  - 叙拉古跨站 role
  - 企鹅 bond
  - 推王 bond
  - 自动化产线
  - 红松林产线
  - 散件房间
  ↓
用命名 policy 解决冲突
  ↓
剩余房间用散件搜索补齐
  ↓
输出单班 / 三队轮换 / MAA JSON
```

这个设计的优点是：

- 缺一个干员时可以自然 fallback。
- 新增策略时不用到处打补丁。
- 同站、跨站、全局资源不会混在一起。
- 人工工具人表继续在贸易等适用领域发挥缩池作用；不裁剪普通排班制造候选。
- 输出结果更接近“懂基建的人会自然排出来的优解”。

## 8. 一个具体例子：小饼账号

错误前的情况是：

程序太相信固定 registry，导致低优先或不完整组合先占了贸易站/人员，结果但书、可露希尔、龙巫没有进入最终轮换。

健康逻辑应该是：

1. 发现有但书：先生成但书核心候选。
2. 叙拉古成员不设“完整链”门槛；八幡、伺夜、贝洛内按各路径实际效率自然竞争，八幡与单个贸易消费者的极端组合也允许胜出。
3. 发现有可露希尔：生成可露核心候选，黑键有就优先，没有就找其他高效工具人。
4. 发现巫恋和龙舌兰：生成龙巫候选，beta 裁缝有就用，没有就 alpha/普通工具人。
5. 再考虑企鹅、推王、普通散件。

这样小饼账号自然会得到：

- 某班有但书贸易站。
- 某班有可露希尔贸易站。
- 某班有巫恋 + 龙舌兰贸易站。

这不是补丁，而是核心策略顺序正确后的自然结果。

## 9. 公孙长乐可以怎样参与

最有价值的输入不是代码，而是策略表。

建议按下面格式给出每个体系：

| 字段 | 例子 |
|------|------|
| 体系名 | 但书核心 / 可露核心 / 龙巫 / 企鹅德狼 |
| 类型 | 全局资源 / 跨站 / 同站 bond / 核心优先 / 散件 |
| 必须干员 | 但书 |
| 优先队友 | 伺夜、贝洛内、高效订单工具人 |
| fallback | 缺伺夜时用最高效率工具人 |
| 互斥 | 但书不和巫恋、可露同站 |
| 优先级 | 单贸龙门币：可露 required；多贸含龙门币：但书 required；全源石：两者关闭；其余候选再按体系与最终效率比较 |
| 锚点效率 | 公孙工具人表中的等效效率 |
| 适用布局 | 243 / 252 / 333 / 特殊说明 |

程序最需要的是这种“策略知识”。只要策略足够清楚，代码应该负责自然搜索和冲突消解。

## 10. 一句话总结

这个项目理想上不是“写死一堆组合的排班器”，而是：

> 用技能真源算机制；普通制造从全部合法候选自然搜索，其余领域按各自策略生成候选。
