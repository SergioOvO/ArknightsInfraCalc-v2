# 基建练卡推荐规则

> 文档角色：canonical
> 生命周期状态：current
> 领域键：advice.training
> 当前真源：self
> 摘要：定义人工规则表、operbox 确定性过滤与 RAG 解释边界的基建练卡推荐合同

## 产品边界

练卡推荐回答：给定玩家 `operbox`，当前应培养谁、先获取谁、哪些组合因核心未齐而暂缓。

职责分层：

```text
体系分析文档 / 技能资料
  = 完整语义、核心与挂件关系、社区术语和解释依据

data/training_recommendations.json
  = 人工维护的结构化规则索引，不承担最终用户文案

确定性过滤器
  = 按 operbox 判定 train / acquire_then_train / blocked / ready / review

RAG
  = 基于过滤结果和体系原文组织回答，不能新增候选或改写判定
```

硬边界：

- 输入第一版只使用 `operbox` 当前字段：拥有状态、精英化、等级、稀有度。
- 不根据设施数量、布局、生产目标或投入意愿做过滤。
- 目标练度由规则表显式给出，程序和 RAG 不得自行推断。
- 推荐只评价基建投入，不声称优先于战斗、主线或其他玩法培养。
- 自然语言只能解释结构化结果，不能新增干员、改变目标练度或修改优先级。

## 规则类型与作用范围

规则类型（`kind`）描述推荐关系的绑定强度：

| kind | 含义 | 硬核心准入 |
|---|---|---|
| `system` | 大体系，如迷迭香、红松林 | 有 |
| `combo` | 小组合，如同站/跨站二人组、银灰中枢与喀兰/孑 | 有 |
| `standalone` | 独立效率散件，如 40% / 35% / 30% / 25% 档 | 无 |
| `soft_combo` | 弱绑定组合，如红云、水月；保留组合关系但不强绑队友 | 无 |

作用范围（`scope`）描述设施语义，不是规则类型：

```text
same_station | cross_station | control_center | independent
```

同一干员可以同时命中多条规则。某条规则被阻塞，不能全局禁止该干员因散件或其他已满足规则获得推荐。

## 角色、准入与行动

成员角色（`role`）只描述组合内结构位置：

```text
core | important | hanger | independent
```

P0 / P1 / P2 是行动优先级，不能与角色互相替代。

硬核心准入只适用于 `system` 与 `combo`：

```text
必需核心全部已拥有，且每个 pick_one 核心槽至少有一个已拥有候选
  -> 规则准入
否则
  -> 规则未准入
```

核心已拥有但未达目标练度，只是训练缺口，不是“核心未齐”。

行动类型：

| action | 含义 |
|---|---|
| `train` | 已拥有，当前可直接培养 |
| `acquire_then_train` | 未拥有，但允许建议获取后再培养 |
| `ready` | 已拥有且已达目标 |
| `blocked` | 规则当前未准入，暂无可执行培养建议 |
| `review` | 规则待人工复核，不能伪装成确定推荐 |

准入后的培养规则：

```text
核心未齐
  -> 该规则不产生 train
  -> 已拥有的核心、重要成员、挂件全部暂缓
  -> 缺失核心若允许获取，可输出条件性 acquire_then_train 计划

核心已齐
  -> 已拥有且未达标的核心 / 重要成员 / 挂件进入 train
  -> 未拥有但允许获取的成员进入 acquire_then_train
```

`standalone` 与 `soft_combo` 没有硬核心：

```text
已拥有且未达标 -> train
未拥有且允许获取 -> acquire_then_train
队友存在与否不影响 soft_combo 的独立成员推荐
```

## 获取策略

默认不建议未拥有干员。例外：

```text
2 / 3 / 4 星：默认允许 acquire_then_train
指定赠送五星白名单：当前至少包含苍苔
其他未拥有干员：不产生培养建议
```

获取策略应作为规则或全局策略显式记录，不能由 RAG 根据“看起来容易获得”自行推断。

## 人工规则表

规则载体：`data/training_recommendations.json`。

它是推荐层真源，不是机制真源，不能替代 `skill_table` / `operator_instances` / `MECHANICS_REGISTRY.csv`。

概念 schema（版本 2）：

```json
{
  "version": 2,
  "acquisition_policy": {
    "default_rarity_le": 4,
    "named_exceptions": ["苍苔"]
  },
  "rules": [
    {
      "id": "example_rule",
      "kind": "system",
      "scope": "cross_station",
      "label": "示例体系",
      "source_system_id": "optional_base_system_id",
      "admission": {
        "required_core": ["干员A", "干员B"],
        "pick_one_core": [
          {
            "label": "任选核心槽",
            "candidates": ["干员C", "干员D"]
          }
        ]
      },
      "members": [
        {
          "operator": "干员A",
          "role": "core",
          "target": {
            "elite": 2,
            "level": null,
            "skill_id": "optional_skill_table_id",
            "skill_name": "可选技能名"
          },
          "priority": "P0",
          "acquisition": "owned_only",
          "benefit": {
            "facility": "trade",
            "efficiency_tier": "system_core"
          }
        }
      ],
      "evidence": [
        {
          "path": "docs/或 vault 路径",
          "heading": "对应标题"
        }
      ],
      "review": {
        "status": "confirmed",
        "conflicts": []
      }
    }
  ]
}
```

字段约束：

- `target.elite` / `target.level` 是程序判断门槛；第一版 operbox 不提供技能专精，因此不把 skill rank / mastery 作为过滤条件。
- `skill_id` / `skill_name` 用于来源核对与 RAG 检索，不能代替目标练度。
- `benefit` 只存结构化收益事实，例如设施、效率档位、上限或仓容效果。
- `evidence` 指向体系分析文档或技能资料的具体标题/段落。
- `review.status` 取值：`confirmed` / `needs_review`。
- **不保存面向用户的最终推荐长句 `message`。** 规则表是结构化索引，不是文案库。
- `standalone` 不得声明硬核心；`soft_combo` 不得把普通队友写成硬核心；挂件不得进入必需核心槽。

维护流程：

```text
阅读技能与体系文档
  -> 确认成员角色与硬核心
  -> 确认实际目标练度
  -> 确认获取策略与收益标签
  -> 记录 evidence
  -> 标记 confirmed 或 needs_review
```

`data/standalone_roster.json` 只是搜索候选池，不证明值得培养。散件规则必须人工筛选后写入推荐表，且不得反向改变 solver 候选集合。

## 确定性过滤器

过滤器读取规则表和 `operbox`，输出账号专属推荐包，而不是一串干员名。

输出分区：

```text
now          当前可直接执行的 train
conditional  获取缺失干员或满足前提后的 acquire_then_train
blocked      暂无执行路径的缺核心规则
ready        已达标背景事实（可折叠）
review       待复核规则命中
```

多规则合并：

```text
同一干员只生成一条展示记录
display_priority = 全部有效命中中的最高优先级
matches[] 保留全部规则、角色、收益、evidence
被阻塞规则不能压掉其他规则的有效 train / acquire_then_train
```

条件性计划示例：

```text
规则核心 = A + B，账号只有 A，B 为低星可获取

conditional:
  先获取并培养 B
  完成后再培养 A 与已拥有挂件

blocked:
  当前体系尚未准入，不产生 now.train
```

## RAG 解释层

RAG 输入是过滤器输出的推荐包，外加按 `evidence` / operator / rule 检索到的体系原文。

程序或固定模板先生成事实骨架：

```text
P0：某干员，当前精一，目标精二。
所属：红松林；角色：核心。
体系核心已齐，因此进入当前可培养。
```

RAG 再补充：

- 为什么有收益
- 属于哪个体系或小组合
- 缺核心时为什么暂缓
- 获取后应如何继续

硬禁止：

- 从推荐包外新增干员
- 改变目标练度或优先级
- 把 `conditional` 写成当前可直接培养
- 把 `blocked` 写成已成立体系
- 把 `review` 写成确定事实
- 自行发明不符合社区习惯的缩写或体系简称
- 把内部 reason/code 直接当作用户文案

术语优先采用体系分析文档原文。若社区叫法仍不统一，可另建小型术语表，不得把长篇文案回写进规则表。

已知实现注意：

- 规则表不再使用面向用户的 `message`；解释应引用 `evidence` 原文。
- 当前 `conditional` / `blocked` 会展开全局未相关项，噪声偏大；产品展示层可折叠，过滤器语义以本文为准。
- 同一干员可能因不同规则同时出现在 `ready` 与 `review`；合并策略仍见实施计划开放项。
- 伪 RAG 解释层尚未接入；当前 `advice` 只输出确定性 JSON。
