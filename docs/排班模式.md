## 1. 模式分层

基建换班按实际执行逻辑分为三个显式模式：

| 模式 | 触发与输出 | 当前范围 |
|---|---|---|
| `auto_rotation` | 游戏内置自动轮换；按组合耗尽触发，不输出 MAA/Mower 计划 | 后续独立设计 |
| `timed_rotation` | MAA 或人工按固定时间/相对间隔执行完整班次表 | 当前优先模式 |
| `mower_dynamic` | Mower 根据实时心情动态决定换班、宿舍与替补 | 暂缓，未经用户恢复不得推进 |

三个模式只共享干员池、机制结算、组合合法性、体系规则和心情/最长工作时间等基础能力。它们不共享触发条件、替补关系、宿舍责任、菲亚梅塔范围、班次概念或输出责任，不得用“统一大排班结构 + 最后转换输出”重新合并。

## 2. 默认模式

用户未声明模式时进入 `timed_rotation`。MAA 与人工手动换班共用定时换班求解规则；具体班次、峰值口径、菲亚梅塔、无人机和宿舍细则由 [定时换班](定时换班.md) 裁决。

当前 `plan` / `layout team-rotation` 的 ABC 三队实现事实见 [排班轮换](SCHEDULE_ROTATION.md)。实现参考不得反向裁决未来模式语义。

## 3. 自动轮换边界

- 只替换心情耗尽的组合；未耗尽组合不随完整班次一起更换。
- 每个主力组合原则上需要独立替补，跨设施关系应受限。
- 默认不主动规划菲亚梅塔，不把自动轮换作为 MAA 或 Mower 输出。
- 正式实现前需要独立 canonical、输入契约和验收案例。

## 4. Mower 边界

- 不使用二班、三班、四班作为核心模型。
- 不复用定时换班求解器，只复用基础能力。
- 回满、耗尽、红脸 0 心情、宿舍黑名单、实时监控和动态替补属于 Mower 专属规则。
- Mower 下菲亚梅塔不套用定时换班的目标规则。
