# 定时换班（公孙长乐 × InfraCalc）

> 文档角色：canonical
> 生命周期状态：current
> 领域键：schedule.timed
> 当前真源：self
> 摘要：裁决定时换班的班次、目标和输出规则

> 适用逻辑：`timed_rotation`
> 适用工具：MAA / 人工手动换班
> 不适用：游戏内置自动轮换 / Mower 动态换班
> 模式总纲：[排班模式](排班模式.md)
> 当前实现参考：[排班轮换](SCHEDULE_ROTATION.md)
> 相关规则：[Fiammetta.md](Fiammetta.md)、[CONTROL_CENTER_ASSIGNMENT.md](CONTROL_CENTER_ASSIGNMENT.md)

## 1. 一句话结论

定时换班是面向 **MAA 与人工手动换班** 的排班模型。它以玩家声明的换班间隔和班次数量为输入，输出固定班次表、菲亚梅塔动作、无人机动作与简化宿舍安排。

定时换班与 Mower 动态换班不能混用求解。定时换班下存在明确的二班、三班、四班；Mower 动态换班下不存在固定班次。

定时换班当前优先落地，默认入口如下：

```text
用户未声明换班逻辑时 → 默认 timed_rotation
```

该默认入口不需要在每次输出中专门提示。

后续 CLI / UI 可显式提供模式设置，但具体输入命名由后端开发确定：

```text
auto_rotation / timed_rotation / mower_dynamic
```

## 2. 适用范围与边界

### 2.1 适用对象

定时换班适用于：

- MAA；
- 人工手动换班；
- 规律循环换班；
- 用户明确声明班次长度、班次数量或作息需求的场景。

MAA 与人工换班共用同一套定时换班规则，可以使用统一求解器。

二者不应过度拆成两套业务逻辑。实际求解时，核心输入都是：

```text
班次数量 + 操作间隔 / 相对间隔 + 是否有固定作息限制
```

人工换班可以是每天 2 次、3 次或 4 次操作；MAA 也可以执行非 24h 周期但规律的相对间隔循环。求解器应按时间间隔与班次结构处理，而不是先按“人工 / MAA”强行拆分规则。

### 2.2 不适用对象

定时换班不适用于：

- 游戏内置自动轮换；
- Mower 动态换班；
- 回满 / 耗尽 / 红脸 0 心情上班等 Mower 特有逻辑；
- 宿舍黑名单与实时心情监控；
- 多个主力组合共用一个替补组合的动态替换关系。

这些属于其他使用逻辑，不应混入定时换班求解器。

### 2.3 与自动轮换的关系

自动轮换后续可视为具备特殊要求的二班制单独落地，形成：

```text
docs/自动轮换.md
```

但自动轮换不沿用本文的全部结构。本文只说明边界，不进入游戏内置自动轮换排班模型。

### 2.4 与 Mower 的关系

Mower 动态换班暂缓。本文只明确边界：

```text
Mower 不套用定时换班求解器。
```

尤其是菲亚梅塔：Mower 下不套用定时换班的但书、巫恋 + 龙舌兰、清流、可露希尔等目标规则。

## 3. 默认优化口径

定时换班默认采用：

```text
峰值优先
```

含义：

1. 先确定账号当前可组成的最高效率主力；
2. 主力角色优先放到最先用的班次；
3. 不因为替补弱、宿舍不足而先降低主力效率；
4. 若某个主力组合无法适配用户声明的班次长度，则直接放弃该组合或调整班次。

长期平均产出优先可作为后续高级模式，但不能用匿名权重混入默认规则。若未来支持，需要单独显示峰值损失与长期收益。

## 4. 输入与输出

### 4.1 输入

定时换班至少需要：

- 布局蓝图；
- 干员盒子与练度；
- 用户声明的换班需求。

用户声明可包括：

- 二班 / 三班 / 四班；
- 每班时长，如 `12/12/12`、`8/8/8`、`12/6/6`、`8/8/4/4`、`7/5/7/5`；
- 资源倾向，如龙门币 / 经验 / 赤金 / 搓玉；
- 是否允许菲亚梅塔；
- 是否有固定作息窗口。

若用户未声明换班逻辑，默认进入定时换班。默认可按 `12/12/12` 或 `12/6/6` 等常见间隔尝试。

### 4.2 输出

本文讨论的是规则和边界，具体成果格式由后端开发实现。

按规则层理解，定时换班应能表达：

- 每个班次的完整房间安排；
- 每个班次的持续时间或相对间隔；
- MAA 可用计划；
- 人工可读时间表；
- 菲亚梅塔动作；
- 无人机动作；
- 简化宿舍安排；
- 放弃的高消耗角色 / 组合及原因；
- 与用户声明目标的差异说明。

## 5. 生成顺序

定时换班生成顺序为：

```text
主力编制 → 最小换班组 → 菲亚梅塔 → 排班方案 → 宿舍与宿管
```

该顺序只适用于定时换班。

### 5.1 主力编制

主力编制是账号当前能组成的最高效率工作状态，包含生产与功能设施。

规则：

- 默认峰值优先；
- 主力角色放到最先用的班次；
- 主力先独立确定；
- 不因替补弱、宿舍不足而先降低主力效率；
- 特殊高消耗角色若无法适配当前班次，则直接放弃该角色 / 组合。

这里的“放弃”指在定时换班方案中不采用该角色 / 组合；不转入 Mower 动态逻辑，也不强行设计红脸 / 回满等复杂心情规则。

### 5.2 班次时间长度与心情计算

定时换班中，班次时间长度与心情计算有两种入口。

#### 用户已声明时间间隔

如果用户直接安排了班次时间间隔，则按用户声明的班次长度过场。

此时心情计算的用途是：

```text
判断主力或特殊角色能否支撑该班次长度
```

如果心情消耗不满足要求，则作为放弃某些高消耗角色或组合的依据，例如歌蕾蒂娅等特殊角色。

#### 用户未声明时间间隔

如果用户未安排时间间隔，则逻辑顺序为：

```text
主力组合检索
→ 估算主力心情消耗和最长工作时间
→ 形成某个主力组合持续工作的最大工作时间
→ 寻找替补组合
→ 完成排班轮换方案
```

这里的时间应理解为主力实际上班时间。

在三班及以上班次中，主力可能持续两班甚至更多班次才下班。因此，判断某个主力能否工作时，应看它到下班之前的完整连续工作时间，而不是只看单个班段长度。

### 5.3 最小换班组

最小换班组是必须同时上下班的干员集合。

绑定条件包括：

- 同站组合拆开后核心机制失效；
- 跨设施体系链要求成员同时在岗；
- 某名干员上下班会改变其他成员效率或心情消耗；
- 暖机、心情区间或派系计数要求同步处理。

定时换班下，跨设施体系默认同上同下，体系一般不建议拆开分别下班。

没有强依赖的干员可以分布到不同班次或独立替换，但不应把这种独立替换扩展成 Mower 的动态触发逻辑。

定时换班中的最小换班组用于生成完整班次，不用于 Mower 动态触发。

### 5.4 菲亚梅塔

定时换班可以正常使用菲亚梅塔，规则遵循 `docs/Fiammetta.md`。

核心原则：

```text
让某个高收益核心多上一班 / 持续上班
```

默认时机：

```text
order = pre
```

常规目标优先级由 `docs/Fiammetta.md` 维护，本文只引用：

```text
但书（二发电布局，1级或者2级贸易站）
> 巫恋 + 龙舌兰（三级贸易站）
> 但书（三级贸易站）
> 清流（三个贸易站布局）
> 可露希尔
```
- “巫恋 + 龙舌兰要吃一起吃”指在同一完整循环内分别使用，而不是必须在同一次换班点连续使用。
注意：该优先级只适用于 MAA / 人工定时换班，不适用于 Mower。

菲亚梅塔使用后，应安排菲亚梅塔进入宿舍休息，并强制在宿舍一号位。这不是游戏逻辑硬要求，而是为了降低执行工具误操作的建议。

#### 菲亚梅塔是否形成独立班次

只要使用菲亚梅塔，就视作一次班次切换。

区别在于：

1. 如果使用菲亚梅塔与常规换班间隔重叠，则两者在同一班次中表达；
2. 如果使用菲亚梅塔与角色换班不重叠，本次操作只有使用菲亚梅塔，也仍然新起一个班次；
3. 只有菲亚梅塔动作时，其他人员前后可以无差别。

#### 菲亚梅塔对组合的影响

使用菲亚梅塔会延长主力角色工作时间，从而导致替补角色或方案调整。

典型规则：

- 给巫恋和龙舌兰时，固定视作巫恋 + 龙舌兰持续工作；第三人裁缝位置由两名裁缝轮流上班，需要进行换班；
- 给但书、可露希尔、清流等单人角色时，该角色持续上班；队友位置根据剩余 1 个或 2 个空位安排替补；


在巫恋、龙舌兰、但书、可露希尔都有，且 243 基建布局使用菲亚梅塔的条件下，可以引入：

```text
主力 1 组：巫恋 + 龙舌兰 + 但书
主力 2 组：可露希尔
```

这属于 243 使用菲亚梅塔时的特殊启用路径。

### 5.5 排班方案

排班方案根据用户声明的班次时长生成。

定时换班中的班次是真实输出对象：

- 每个班次有完整房间安排；
- 每个班次有固定执行时间或相对间隔；
- MAA 需要明确执行时间或循环顺序；
- 人工换班输出人类可读时间表。

定时换班不引入：

- 回满上班；
- 耗尽换班；
- 红脸 0 心情上班；
- 宿舍黑名单；
- Mower 的动态替补选择。

### 5.6 宿舍与宿管

定时换班宿舍设计保持简化。

规则：

- 宿舍主要用于下班干员正常恢复；
- 除特殊消耗异常角色外，不专门设计复杂单回 / 群回规则；
- 条件允许时放 1-2 名宿管即可；
- 宿管优先群回；
- 不默认要求每个干员回满；
- 不默认设计红脸 0 心情上班；
- 不默认设计宿舍黑名单。

如果某角色或组合依赖精细心情区间才能工作，则优先判断它是否适配当前定时班次；不适配则放弃，而不是把定时换班改造成动态换班。

#### 群体回复宿管优先级

定时换班中，群回宿管优先级为：

```text
流明E2（有自动化组上班）
> 塑心E2（有感知信息上班）
> 流明E2
> 隐德来希E2
> 波登可E1
> 杜林
> 安比尔 = 夜莺E2
```

其余群体回复宿管直接按照宿管技能效果建模安排。

单体回复优先级直接按照宿管技能效果建模安排。

#### 明确使用单体宿管的情况

定时换班在以下条件下明确使用单体宿管：

1. 歌蕾蒂娅作为主力工作，采取 `7/5/7/5` 班次时，歌蕾蒂娅休息时进入的宿舍要求有 1 单体 + 1 群体宿管；
2. 存在心情异常消耗的情况，例如阿罗玛额外心情消耗、巫恋 + 龙舌兰 + 但书一组对但书带来的额外心情消耗，可以给存在额外心情消耗的角色安排 1 单体 + 1 群体宿管。

## 6. 班次解释

### 6.1 二班

二班是固定两套循环班次。

业务定义：

```text
一主一替
```

也就是每个主力组合或主力角色都有自己的替补。不同主力的替补角色必须不同。

适用于：

- 用户明确需要两班逻辑；
- 操作希望简单；
- 用户愿意接受一定效率损失换取低操作负担。

二班不是 Mower 的动态主替关系，也不是游戏自动轮换。

### 6.2 三班

三班是固定三套循环班次。

业务定义主要包括两类：

1. 两套主力共用一套替补，轮流进行休息；
2. 多个换班组错峰运行，最终形成三种主要状态。

常见形式包括：

- `8 / 8 / 8`；
- `12 / 12 / 12`，一个完整周期约 36h；
- `12 / 6 / 6`；
- 其他规律三段式换班。

三班适用于 MAA 和人工定时换班。当前求解器已有三班相关实现基础。

### 6.3 四班

四班不应机械理解为所有岗位都有三套完整替补。

定时换班中，四班主要只在特殊要求下考虑。目前典型情况包括：

1. 三班次中插入菲亚梅塔使用；
2. 深海特殊心情限制。

#### 菲亚梅塔插班

三班次中插入菲亚梅塔，主要是将：

```text
16 / 4 / 4
```

拆分为：

```text
8 / 8 / 4 / 4
```

原因是：`16/4/4` 只能在 16h 班次的前后各使用一次菲亚梅塔，而菲亚梅塔从 0 心情回满需要 12h，中间存在约 4h 没有使用，造成菲亚梅塔资源浪费。

为了充分使用菲亚梅塔资源，可以在中间插入一次菲亚梅塔使用，从而拆成 `8/8/4/4`，提高收益。

#### 深海 7/5/7/5

歌蕾蒂娅相关四班制主要用于深海特殊心情限制。

4 深海的歌蕾蒂娅最多工作约 7.5h。为了多次重复使用歌蕾蒂娅做主力，可以形成：

```text
7 / 5 / 7 / 5
```

该方案主要在 153 布局中实现。

### 6.4 五班及以上

默认不支持。

除非有具体账号、具体体系和明确收益，否则更多班次状态大概率只会降低效率并增加执行复杂度。

MAA 定时换班最多 4 个班次状态是建议上限。超过 4 个状态时，应提醒或警告，而不是默认无声生成。

## 7. 时间表达与安全余量

### 7.1 MAA 时间表达

MAA 最适合 24h 周期，明确每个班次具体执行时间。

对于非 24h 周期但规律轮换的场景，时间表达应采用：

```text
相对间隔
```

非 24h 周期不以每日固定时间点输出执行时间，而是按照 MAA 协议输出提示间隔。

原因：非 24h 周期无法稳定表达为每日固定绝对时间点，但可以形成规律循环。

### 7.2 人工时间表达

人工换班更适合输出人类可读时间表。

人工换班每天最多 3 次操作只是默认建议，不是硬约束。用户明确要求 4 次时可以允许。

实际求解不需要过度区分人工或 MAA，只要根据时间间隔和班次结构输出求解即可。

如果用户给出固定作息，应优先适配作息；若无法适配，应提示需要降低效率、放弃高消耗组合或调整班次。

### 7.3 安全余量

定时换班下，理论换班时间需要安全余量。

安全余量规则：

```text
按更粗粒度取整，最好允许约半小时容错
```

半小时容错是经验余量。实际上绝大多数心情消耗正常的干员不会触发该问题，但不能把理论极限时间直接当作实际换班时间。

## 8. 菲亚梅塔与无人机

### 8.1 菲亚梅塔

定时换班中，菲亚梅塔按 `docs/Fiammetta.md` 执行。

关键规则：

- 菲亚梅塔本人满心情时才可使用；
- 固定每小时回复 2 点；
- 默认换班前使用；
- 目标按收益率与公孙给定优先级选择；
- 使用后菲亚梅塔进入宿舍休息；
- 菲亚梅塔位于宿舍一号位置是降低误操作的建议；
- 不写完整宿舍恢复模拟和全局心情最优。

### 8.2 无人机

定时换班中，无人机按 `docs/DRONES.md` 执行。

关键规则：

- 一般有就用；
- 两次使用间隔建议在 13h 以内；
- 默认换班前使用；
- 根据用户资源倾向、赤金供需、钱书比例与贸易乘区决定投向；
- 不用于加速合成玉订单。

## 9. 高消耗与特殊角色

定时换班下，如果特殊高消耗角色经过计算无法适配当前班次，则直接放弃。

这类处理包括：

- 不强行把该角色塞进当前班次；
- 不转成 Mower 动态逻辑；
- 不设计红脸 0 心情上班；
- 不要求完整宿舍精细恢复；
- 在输出中说明放弃原因。

一名高消耗干员是否拖动整组提前下班，取决于该干员在组合中的地位：

- 如果是组合中的核心角色，则提前下班，通常带动整组下班；
- 如果只是普通角色，可以单独找替补；
- 一般建议整个组合直接下班，以避免把定时换班复杂化为动态换班。

后续需要维护一份特殊高消耗角色 / 组合的判定规则或清单。

## 10. 与其他逻辑的区别

### 10.1 与自动轮换

自动轮换可后续作为具备特殊要求的二班制落地，但不是本文模型。

差异：

| 项 | 定时换班 | 自动轮换 |
|---|---|---|
| 触发 | 固定时间 / 相对间隔 | 组合心情耗尽 |
| 替换范围 | 整班或预设班次 | 只替换耗尽组合 |
| 跨设施体系 | 可按班次设计 | 达到三设施及以上默认禁用；多名中枢体系应减少 |
| 菲亚梅塔 | 可正常规划 | 不主动规划 |
| 输出 | MAA / 人工时间表 | 游戏内置自动轮换可行性 |

### 10.2 与 Mower 动态换班

Mower 动态换班暂缓。

差异：

| 项 | 定时换班 | Mower 动态换班 |
|---|---|---|
| 班次 | 有固定班次 | 无固定班次 |
| 心情监控 | 理论估算 + 安全余量 | Mower 实时监控 |
| 替补 | 体现在完整班次 | 主力 / 替补 / 替换关系 |
| 宿舍 | 简化宿舍；特殊时使用单回 + 群回 | 每宿舍一名单回 + 一名群回，黑名单等 |
| 菲亚梅塔 | 使用定时换班规则 | 不给巫恋、龙舌兰、但书、可露希尔；具体目标后续确认 |

## 附录 B：已确认口径

1. 用户未声明时，默认进入定时换班逻辑。
2. CLI / UI 具体命名由后端开发确定，但模式层需要能区分自动轮换、定时换班、Mower 动态换班。
3. MAA 与人工手动换班共用定时换班逻辑。
4. 定时换班默认采用峰值优先。
5. 用户给定时间间隔时，按时间间隔判断心情是否能支撑；用户未给定时间间隔时，先求主力，再估算最长工作时间，再找替补并形成轮换。
6. 人工定时换班默认最多每天 3 次操作，但允许用户明确要求 4 次。
7. MAA 定时换班最多建议支持 4 个班次状态；超过时提醒 / 警告。
8. MAA 非 24h 周期但规律轮换时，时间表达采用相对间隔，不输出为每日固定时间点。
9. 定时换班理论换班时间需要经验安全余量，最好允许约半小时容错。
10. 二班业务定义是一主一替，不同主力可以有不同替补。
11. 三班业务定义是两套主力共用一套替补，或多个换班组错峰运行形成三种状态。
12. 四班主要用于菲亚梅塔插班和深海特殊心情限制，不机械生成第四套低效编制。
