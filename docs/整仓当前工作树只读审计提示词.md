# GPT-5.6 整仓当前工作树只读审计提示词

> 文档角色：current-reference
> 生命周期状态：current
> 当前真源：docs/QUALITY_AND_AUDIT.md；docs/PROJECT_MAP.md；docs/文档生命周期.md
> 摘要：提供整仓当前工作树非 formal、只读、证据校准型代码与文档拓扑审计提示词

本文正文可直接作为 GPT-5.6 的任务提示词。执行时捕获的 global snapshot 才是审计对象；本文不保存某一次会话的实时工作区状态。

## 任务

你位于 ArknightsInfraCalc-v2 仓库根目录。

请对“执行时冻结的整个当前工作树”执行一次证据校准型代码审计。目标是发现具有现实触发条件的正确性、领域一致性、公开接口兼容性、solver 保证、安全、并发和韧性缺陷，并诚实说明所有未覆盖面。同时审计当前文档信息架构能否让人类和 Agent 找到唯一事实 owner，产出独立的文档拓扑报告和后续重建 handoff。

本任务只审计，不修改代码或 tracked 文档。文档重建只能形成 ignored evidence artifact 和后续 writer handoff，不能在本轮直接应用。

## 1. 审计模式与成功标准

本任务是广义的：

```text
NON-FORMAL WHOLE-REPOSITORY CODE AUDIT
```

它不是用户要求对每个具名基建体系进行严格逐项验收，也不自动触发 `formal-audit`。

必须遵守：

- 这不是 diff review。
- 当前 dirty diff 是高风险优先面，不是审计范围边界。
- 审计范围覆盖当前快照中的全部已跟踪产品代码、配置、运行时数据、脚本、测试、current 文档、适用的 accepted ADR，以及非 ignored 的相关未跟踪源码和文档。
- 可以报告当前快照中的存量缺陷。
- 没有历史基线证据时，只能称为 `current defect`，不能称为“新引入的 regression”。
- 整仓不是一个无界审计单元。先盘点，再拆成 bounded audit packages。
- 每个 package 只能有一个可证伪不变量。
- 不要求凑 finding 数量。
- 发现一个问题后继续完成其他 package。
- 没有充分证据的猜测不得进入 findings。
- 只有达到 stop condition 的 `audited` package 才能支撑 no-finding 结论。
- `partial`、`deferred`、`blocked`、`invalidated` package 不能产生 no-finding 结论。
- 所有最终 findings、no-finding surfaces、package 状态和命令证据必须绑定同一个稳定 global snapshot。
- 无法形成统一稳定快照时，最终整体状态必须是 `PARTIAL` 或 `BLOCKED`。
- 文档拓扑结论必须来自已审计的真实路径、owner、命令和生命周期事实，不得以“更好看”或通用模板偏好代替证据。

`COMPLETE` 只表示完成了这里定义的广义非 formal 代码审计，不表示完成了每个具名体系的 formal semantic acceptance。

## 2. 只读与安全边界

审计模式固定为：

```text
TRACKED-SOURCE-READ-ONLY
```

禁止：

- 修改任何 tracked 文件。
- 修改任何非 ignored 的源码、数据、文档或配置。
- stage、commit、amend、reset、checkout、restore、stash、rebase、clean、push。
- 创建或删除 Git worktree。
- 修改 Git config、凭据、远端或外部系统。
- 自动修复 findings。
- 生成补丁。
- 整理、覆盖或回滚用户已有改动。
- 联网、安装依赖、更新 lockfile、调用外部前端仓库或在线漏洞情报。
- 执行会原地重写 tracked data、Bake、release 或 docs 的 generator。
- 运行会写回源码的格式命令。

仅允许在确认被 Git ignore 后写入：

```text
target/codex-runs/<task>/
target/
out/<audit-slug>-*
```

其他要求：

- 最终证据不能只放在 `/tmp`。
- Cargo 命令使用 `--locked` 和离线模式。
- 缓存缺失导致命令无法运行时，登记 verification gap，不得联网补齐。
- 不读取或输出 `.env`、凭据或个人数据正文。
- 日志和报告不得泄露敏感输入。
- 需要验证 generator 时，只能使用 check/dry-run 模式，或复制输入并将全部输出定向到已确认 ignored 的任务目录。
- Rust 格式检查只能运行 `cargo fmt --all -- --check`。
- 临时 repro/harness 必须放入任务专属 ignored 目录，不得注入 tracked source。
- 如果任何审计命令意外改变 tracked 或非 ignored 内容，立即停止相关命令，不回滚，并把它记录为审计安全违规和 drift。最终状态至少为 `BLOCKED`。

首次写入 evidence/output 前：

1. 直接读取 `.gitignore`。
2. 用只读 Git 检查确认目标路径确实 ignored。
3. bootstrap 阶段可裸跑最小的 `git check-ignore` 和 status。
4. bootstrap 命令不能支撑产品结论。
5. wrapper 可用后，必须以 evidence 形式重跑所有用于结论的命令。

所有用于结论的 build、test、CLI、benchmark、format、structure 或 artifact 命令，必须通过 `scripts/codex/run_evidence.sh` 或仓库认可的等价 evidence wrapper 留痕。

### 3.2 Formal Audit 边界

本次广义审计不得对每个具名体系给出以下 formal completeness verdict：

```text
已严格实现
未完整实现
完整生命周期逐项验收通过
```

普通 system package 只审计通用责任边界，例如：

- required admission 由谁保证；
- Plan 是否表达核心、producer、optional 和 degradation；
- same-room/cross-station/in-base scope 是否在正确层保证；
- `shift_bind` 是否只消费已进编成员；
- rotation/export 是否忠实消费既有 Plan；
- Rule、Plan、Execute、Fill、Resolve 的 owner 是否错位。

若调查需要为一个具名体系建立完整领域不变量表、对其完整执行 `select -> plan -> execute -> fill -> resolve -> rotation -> export` 严格验收，或两个适用的 current canonical Markdown 发生冲突，则该 package 已越界进入 `formal-audit`。

此时：

1. 不得把 formal audit 塞回广义 package。
2. 对 system 类问题，完整读取 `docs/SYSTEM_AUDIT_WORKFLOW.md`。
3. 一次只处理一个体系。
4. 提交第一阶段只读审计报告。
5. 遵守用户裁决等待点。
6. 广义审计可以继续不受影响的 package。
7. 最终把该项报告为 `formal-audit-required`。
8. 不得宣称该具名体系已在本轮完成 formal acceptance。

非 system canonical 冲突同样必须阻塞受影响 package 并等待用户裁决，但不要错误套用体系 formal workflow。

### 3.3 真源顺序

真源顺序固定为：

1. 用户当前明确裁决。
2. 对应领域 current canonical Markdown。
3. 代码和生成 help 证明当前实现事实。
4. JSON、CSV、fixtures、tests、Bake 和历史输出只承载或核对实现。

补充规则：

- `docs/TODO/**` 不解释当前产品行为。
- `docs/ARCHIVE/**` 不解释当前产品行为。
- TODO/ARCHIVE 只参与 lifecycle、引用泄漏、状态真实性和任务边界审计。
- `docs/ADR/**` 中 accepted 决策约束架构责任、迁移边界和兼容承诺。
- accepted ADR 不得覆盖领域 canonical。
- 每个 package 必须登记适用的 accepted ADR。
- ADR 中明确标为 future、deferred 或 accepted-but-unimplemented 的能力，不能仅因尚未实现而报 bug。
- 只能检查当前文档是否诚实，以及当前实现是否违反已经生效的边界。

## 4. Capability Gate 与委派

委派前核验当前运行时真正暴露并成功加载的 profile。

建议用途：

- `luna-extractor`：提取稳定字段、日志、表格、failure sets 和 coverage ledger。
- `sol-reviewer`：高风险结论和最终报告的独立反方复审。

必须记录：

- profile 名；
- 任务边界；
- 是否成功加载；
- 返回状态；
- 绑定的 package 和 snapshot ID。

禁止：

- 用 task label、昵称或提示词假装切换了模型/profile。
- 把主 Agent 的第二遍检查称为 independent review。
- 在 profile 不可用时，把全部机械提取和所有 package 顺序回落给默认 Sol，再宣称同等覆盖。

- profile 不可用时，ledger 写 `unavailable`。

- 委派失败、输出截断、未绑定 snapshot 或未绑定 package 的结果不算完成。

## 5. 冻结统一 Global Snapshot

任何代码结论前必须冻结快照。

### 5.1 Snapshot Manifest

记录：

1. 仓库绝对路径。
2. `HEAD`。
3. branch。
4. `git status --short --untracked-files=all`。
5. staged、unstaged、untracked 分类。
6. `git ls-files` 返回的全部 tracked path。
7. `git ls-files --others --exclude-standard` 返回的全部非 ignored untracked path。

为每个 in-scope path 记录：

- 规范化仓库相对路径；
- tracked/untracked 分类；
- 产品/配置/数据/测试/文档等分类；
- 文件类型和模式；
- 字节数；
- 内容 SHA-256；
- tracked deletion 使用 `MISSING`；
- symlink 记录 link 自身，不盲目跟随到仓库外。

对规范化 manifest 计算 SHA-256：

```text
snapshot_id = <HEAD>:<manifest_digest>
```

把 snapshot manifest 和初始 status 保存到 ignored evidence 目录。

规则：

- ignored 的 `target/`、`out/` 和审计 evidence 不进入 source snapshot。
- 若某路径实际 tracked，即使位于这些目录中，也必须进入 snapshot。
- mtime 不能代替内容 hash。
- 每个 package report、finding、命令 run 和 reviewer 结论都必须记录完整 snapshot ID。
- 本文不提供预期 dirty 文件清单；执行时捕获的状态是唯一权威。

### 5.2 Drift 与失效传播

每个 package 开始、结束以及最终 synthesis 前后，都要比较 in-scope content manifest。

任一 in-scope drift：

1. 生成新的全局 snapshot ID。
2. 判断是审计命令导致还是外部并发变化。
3. 不得回滚。
4. changed path 与 package 的 truth、required、consumer 或 proof path 相交时，该 package 立即标为 `invalidated`。
5. 在新快照重跑受影响 package。
6. 新增或删除文件时重新执行 inventory 和 reverse path coverage。
7. `AGENTS.md`、skills、`docs/INDEX.md`、适用 canonical/ADR、Cargo manifests、crate root、公开入口发生变化时，向所有依赖其路由、契约或入口闭包的 package 传播失效。

未直接受影响的 package 只有在以下条件全部满足后，才可 rebind 到新 snapshot：

- 机械证明其全部 truth/required/consumer/proof closure 内容未变；
- 全局路由未变；
- inventory 分类未变；
- package report 明确记录 rebind 依据。

否则必须重跑。

最终报告不得混合不同 snapshot。反复漂移、无法证明失效传播或最终快照不稳定时，整体状态必须为 `PARTIAL` 或 `BLOCKED`。

## 7. Audit Packages 与 Evidence Ownership

Coverage group 只是盘点容器，不能直接标为 `audited`，也不能产生 no-finding 结论。

每个 package 必须只有一个单句、可证伪 invariant。

### 7.1 Package Schema

每个 package 至少包含：

```text
group_id
package_id
single falsifiable invariant
snapshot_id
risk
primary Skill
assigned profile
named production or documentation-consumer entrypoint
applicable canonical
applicable accepted ADR
required paths
consumer paths
proof paths
explicit exclusions
solver guarantee class, if applicable
analysis status
scope gate
render gate
output safety gate, if applicable
stop condition
commands
artifacts
findings
rejected candidates
verification gaps
```

`analysis status` 只能是：

```text
planned
in_progress
audited
partial
deferred
blocked
invalidated
```

`audited` 只表示语义和只读分析 stop condition 已满足。

另行记录：

```text
scope_gate = PASS | FAIL | UNSATISFIED | NOT_RUN
render_gate = PASS | FAIL | NOT_RUN
output_safety_gate = PASS | FAIL | NOT_RUN
```

即使 package 的 analysis status 为 `audited`，只要任一适用的 completion gate 非 PASS，整体也不能为 `COMPLETE`。

以下名称过大，不能直接作为 package invariant：

```text
整个 layout
接口兼容性
安全
并发
韧性
全部 solver
整个 training domain
全部文档
```

### 7.2 每 Package 独立 Evidence Manifest

每个 package 必须拥有独立且不可复用的 task slug：

```text
<audit-slug>-<snapshot-short>-<package-id>
```

每个 package 独立拥有：

- schema-v3 metadata；
- 一个 invariant；
- required/consumer/proof/deferred paths；
- reviewer record；
- manifest；
- runs；
- artifacts；
- rendered report；
- scope-check result。

禁止：

- 多个不变量共享一个 task manifest；
- 后续 package 覆盖前一个 package 的 metadata；
- 用 parent coverage ledger 代替 package manifest；
- 用一个巨大 audit invariant 包住全仓。

每个 evidence run 的 `inputs` 字段都要包含：

```text
package_id=<id>
snapshot_id=<full snapshot id>
```

一个命令若被多个 package 共享：

1. 建立一个单一不变量的 shared-proof package。
2. 命令只归属于该 shared-proof package。
3. consumer package 引用其 manifest/artifact。
4. consumer package 在 ledger 中记录 evidence dependency。
5. 不复制或伪造 command ownership。

## 9. Package 执行协议

每个 package 严格按以下顺序执行：

1. 写出一个可被反例推翻的 invariant。
2. 完整读取该 package 对应的 canonical Markdown。
3. 按需读取 current reference、accepted ADR 和相关 registry/data rows。
4. 从具名 production entrypoint 或 documentation-consumer entrypoint 建立实际可达调用和数据闭包。
5. 标记 compatibility、generator、test-only、legacy、dead 和 reachability-unconfirmed 路径。
6. 不能靠文件名、Glob 或文本命中宣称闭包完整。
7. 追踪 required -> producer/owner -> consumer -> proof path。
8. 检查合法调用方、错误路径、边界输入、状态转换和输出契约。
9. 构造能够改变 owner 判断的最小反例。
10. 对 system 类问题区分 admission/activation/bind、eligibility/required placement、selection/settlement、真实输出/偶然 top hit。
11. 先证伪候选 finding。
12. 读取完整函数和必要上下文。
13. 检查全部直接调用方和消费者。
14. 检查现有测试以及相反证据。
15. 确认生产路径或真实文档消费者可达、触发条件现实、canonical 没有许可该行为。
16. 所有命令使用 evidence wrapper。
17. 检查完整 log、status、exit code 和 artifact。
18. 不把命令 PASS 当领域真源。
19. package 结束时做 drift check。

达到以下 stop condition 后才可标 `audited`：

- applicable truth sources 已确认；
- required/consumer/proof paths 已追到明确边界；
- 入口闭包边界已说明；
- 候选 findings 已由合法调用方和反例检验；
- 相反证据已检查；
- 未覆盖路径已登记；
- evidence 绑定当前 snapshot；
- reverse path map 已更新。

“读完目标文件”不是 stop condition。

发现旁支问题时，登记到对应 package 或新建单一不变量 package，不得在当前 package 混入第二个不变量。

不得为单一 operbox、room id、Shift index、fixture 或当前 top hit 编造普遍结论。

### 9.1 文档拓扑专项协议

文档拓扑不是单一 package。按 G19 的责任边界分别审计，再合成报告。

必须完整读取 `docs/文档生命周期.md`，并以当前角色、状态、唯一领域 owner 和生成索引规则为本仓库约束。外部成熟方法只能用于组织读者需求，不能覆盖仓库 canonical。

至少检查：

```text
README.md
AGENTS.md
docs/INDEX.md
docs/PROJECT_MAP.md
docs/OVERVIEW.md
docs/GONGSUN_RUNTIME_OVERVIEW.md
docs/ARCHITECTURE_TOUR.md
docs/QUALITY_AND_AUDIT.md
docs/MAINTENANCE_MODE.md
docs/SYSTEM_AUDIT_WORKFLOW.md
docs/INFRA_CLI.md
docs/SCHEDULE_ROTATION.md
docs/ADR/*
.agents/skills/*/SKILL.md
crates/infra-core/src/lib.rs
```

按实际 owner 和调用方继续扩展，不因该清单存在就假定覆盖完整。

建立一份事实 owner 矩阵，至少区分：

```text
产品定位与 quickstart
领域语义
当前产品能力与明确非目标
代码/命令/data owner
端到端动态调用链
Agent 任务路由与硬门禁
工作流细节
公共 Rust API
持久架构决策
活动工作与历史材料
生成或机械校验的事实
```

每项事实只能指定一个 authoritative owner；其他文档只能链接、压缩摘要或作为具名 consumer。发现多份文档分别声称自己是 owner 时，报告冲突和真实消费者，不自行选择语义真源。

按读者需求区分：

- tutorial/quickstart：让新用户第一次跑通；
- how-to：完成一个具体任务；
- reference：精确列出命令、schema、owner 和 API；
- explanation：解释领域模型、架构和取舍。

该分型用于发现一份文档混合多种读者任务的风险，不要求为了套目录模板而批量移动文件。

架构说明按层级检查：

```text
System context
Rust workspace/container boundary
infra-core component boundary
关键 dynamic flows：plan、plan.compute、layout/rotation/export
```

不要求手写完整 code-level 图；能由源码、Cargo metadata、CLI help 或 generator重建的易变事实应优先生成或机械校验。

建立独立的 `documentation-topology-synthesis` package，其唯一不变量是：报告和 handoff 中的每项确定性判断及后续改写动作都能追溯到一个 `audited` G19 package。该 package 只消费 G19 package reports，不替代底层审计。另将“artifacts 不得成为或冒充 current truth”登记为 output safety gate 和 non-goal，不作为第二个 package invariant。

必须在该 package 的 `target/codex-runs/<task>/reports/` 下产出两个 ignored artifacts：

1. `documentation-topology-report.md`：当前职责、重复事实、错误 owner、失真/不可验证表格、读者路径、Agent 导航失败点和保留良好边界。
2. `documentation-rewrite-handoff.md`：供后续独立 `arknights-quality` writer 使用的有界重建任务，不是本轮补丁。

对具有 audited 依据的候选文件，handoff 至少列出以下字段；对 `not adjudicated` 文件，只能列出 `path`、缺失证明、受影响事实、阻塞 package 和后续调查入口，不得填写改写动作、proposed section outline、acceptance checks 或 migration dependencies：

```text
path
target reader
single responsibility
current authoritative owners
confirmed facts to retain
duplicated or misplaced facts to remove
facts to replace with links
facts suitable for generation/checking
proposed section outline
required source/consumer/proof paths
acceptance checks
non-goals
ordering and migration dependencies
```

对具有 audited 依据的 `README.md`、`AGENTS.md`、`docs/INDEX.md`、`docs/PROJECT_MAP.md`、`docs/OVERVIEW.md`、`docs/ARCHITECTURE_TOUR.md` 和 crate-level rustdoc，分别裁决应为 `keep | trim | rewrite-in-place | split | generate | unchanged` 中哪一种并给出证据；其余候选文件只能报告 `not adjudicated`。不得创建长期并存的“新版 README/AGENTS/PROJECT_MAP”。

仅基于 audited packages 为后续 writer 设计 8 至 12 个 fresh-agent 导航验收场景。每个场景记录用户意图、期望首读、唯一 owner、生产入口或 documentation-consumer entrypoint、验证入口和允许的最大文档跳转数，至少尝试覆盖机制修改、solver 保证、Plan admission、CLI/Worker、排班/export、练卡推荐、数据生成和 docs lifecycle。无法由 audited package 证明唯一 owner、生产入口或 documentation-consumer entrypoint、验证入口的场景不得补写预期结构，只登记为具名 verification gap；若不足 8 个已验证场景，handoff 必须明确报告不足且整体不得为 `COMPLETE`。

如果相关文档 package、代码 owner 或命令入口不是 `audited`，报告和 handoff 只能登记缺失证明、受影响事实和后续调查入口；不得给出 `keep | trim | rewrite-in-place | split | generate | unchanged` 裁决、section outline 或其他改写建议。

本轮禁止：

- 修改 tracked 文档；
- 直接生成可冒充 current truth 的平行“新版”文档；
- 仅按篇幅或审美要求重写；
- 把 Agent memory、审计报告或 handoff 升格为领域真源；
- 用 crate rustdoc 复制领域公式；
- 让 README、AGENTS 或 PROJECT_MAP 重新成为全仓事实汇总。

## 11. Solver Assurance 专项规则

先枚举具名生产 solver 入口，再建立可达闭包。

只对闭包中的空间变换逐项分类：

```text
hard_constraint
safe_reduction
search_heuristic
policy_restriction
approximation
```

需要分类的路径包括：

- eligibility；
- candidate generation；
- pruning；
- decomposition；
- objective；
- tie-break；
- top-K；
- Bake；
- cache；
- fallback。

另表登记：

```text
test-only
generator
legacy
dead
reachability-unconfirmed
```

未证明入口闭包完整时，只能说 `audited entrypoints: ...`，不得说 `all solver paths audited`。

必须核对：

- hard constraint 是否来自 canonical，而非 priority、白名单或 current top hit；
- safe reduction 的适用前提、完整目标保持和组合兼容性；
- heuristic 是否只改变顺序，且 exact fallback 能恢复完整候选；
- policy 是否为用户具名限制；
- approximation 是否报告停止原因、fallback 和降低后的结果状态；
- candidate 自身是否进入 workforce/context；
- identity 去重是否稳定；
- producer/consumer 是否在正确 layout snapshot 求值；
- 多目标是否显式使用词典序或合法分阶段优化；
- 匿名加权是否真的满足高层权重覆盖全部低层可能贡献。

结果状态不得混淆：

```text
EXACT_OPTIMAL
POLICY_OPTIMAL
BEST_FOUND
UNKNOWN
TIME_LIMIT
INFEASIBLE
NO_CANDIDATE_UNDER_POLICY
```

candidate generation、pruning、decomposition 和 Bake 应按风险使用：

- 小实例 oracle；
- 关闭 reduction 的 differential；
- fresh-vs-cache；
- fresh-vs-Bake；
- fallback；
- 等价证明。

做不到时登记 assurance gap 或 verification gap，不能冒充全局最优证明。

定义：

- `ASSURANCE GAP`：产品或实现声称高风险保证，但设计或证明义务本身不足。
- `VERIFICATION GAP`：本次审计因权限、profile、环境、时间或成本未完成验证。

同一缺口不得重复记为两条。

## 12. 命令 Evidence 与 Dirty-Tree 特例

所有 rerun 必须保留，不得覆盖失败。

候选 evidence 类别：

### Structure

- repository facts；
- docs inventory/lifecycle；
- data/schema checks；
- `git diff --check`；
- snapshot/closure hash。

### Format

```text
cargo fmt --all -- --check
```

### Build 与 Test

- 离线、locked workspace build；
- targeted tests；
- 完整 workspace suite；
- Python tooling tests。

### CLI

- `verify --all`；
- 真实 `plan`；
- `layout team-rotation`；
- `layout eval`；
- `advice`；
- JSON-line `ping`；
- `plan.compute`；
- legacy request；
- malformed/error cases。

真实 `plan` 必须使用任务专属输出：

```text
--profile-out out/<audit-slug>-profile.json
--maa-out out/<audit-slug>-maa.json
```

### Bake

- 只读 validate；
- fresh-vs-Bake；
- ignored 临时副本；
- 不得重写 tracked catalog。

### Performance

只有 finding 或 assurance 结论涉及资源和复杂度时才运行有界 benchmark，并记录输入、seed、time limit、fallback 和 observed result status。

命令失败必须区分：

- product failure；
- pre-existing failure；
- environment failure；
- tooling failure。

日志截断、状态缺失或 failure set 无法解析时，不能判 PASS。

### 12.1 Scope Checker 强制门禁

`check_task_scope.py` 仍是 mandatory completion gate，必须对每个最终 package manifest 实际运行。

执行时先核实 checker 是否支持“审计起始 dirty snapshot 减法”。若仍会合并 `base_sha..HEAD`、unstaged、staged 和 untracked，且没有 baseline subtraction，它就无法区分用户开始前已有 dirty paths 与审计者新增 changed paths。

必须：

- 对每个 package 使用真实、单一不变量 metadata；
- 如实填写 required/consumer/proof/deferred；
- 如实填写 reviewer record；
- 实际运行 checker；
- 记录 PASS、FAIL、UNSATISFIED 或 NOT_RUN。

禁止：

- 使用 `--committed-only HEAD..HEAD` 制造空集；
- 手工伪造 empty changed paths；
- 把所有用户已有 dirty paths 塞进 package scope；
- 把用户改动冒充本 package 改动；
- 扩大 manifest scope 来制造 PASS。

起始/最终 content manifest 和 drift ledger 用于额外区分 auditor changes，但不能取代仓库 scope gate。

只要任一 required package 出现 `scope_gate != PASS`，整体 `COMPLETE` 就被禁止。

若权威起始工作树 dirty 且 checker 没有经过验证的 baseline-aware 模式，则相关 package 的 `scope_gate = UNSATISFIED`，整体最高状态为 `PARTIAL`。用户 waiver 可以豁免本次 scope checker 的执行要求，但不得把 `UNSATISFIED` 改写为 `PASS`，也不得据此授予 `COMPLETE`；只有仓库实际提供并验证 baseline-aware checker 后，dirty-snapshot 审计才可满足该 COMPLETE gate。

最终每个 package 必须报告：

```text
check_task_scope: PASS | FAIL | UNSATISFIED | NOT_RUN
```

不得写 `N/A`。

### 12.2 Evidence Renderer

每个 package 必须单独运行 `scripts/codex/render_evidence.py`，并记录：

```text
render_gate = PASS | FAIL | NOT_RUN
```

任一 required package renderer 非 PASS 时，整体不能为 `COMPLETE`。

### 12.3 Full-Suite 与 Regression 措辞

当前 snapshot 可复现的问题只能称为 `current failure` 或 `current defect`。

只有存在明确、可验证且早于当前变化的 baseline，才能称为 `regression relative to <named baseline>`。

“无新增 full-suite failure”必须：

- 比较完整测试全名集合；
- 使用真实 baseline；
- 通过 `compare_test_failures.py`；
- 报告 additions、removals、unchanged。

禁止：

- 把当前运行和自身比较；
- 把本轮第二次运行当历史 baseline；
- 只比较失败数量；
- 在没有 baseline 时声称“无新增回归”。

没有历史 baseline 时，可以报告当前 suite 的通过、失败集合和稳定性，但不得作 regression 结论。

### 12.4 Dependency Security 限制

离线依赖审计只能检查 `Cargo.lock`、features、供应链配置和具有明确时间戳的本地 advisory database。

如果没有可复现且时效明确的漏洞库，最终必须写：

```text
external vulnerability intelligence not verified
```

不得声称 `no known vulnerabilities`。

## 13. Finding 门槛

只报告同时满足以下条件的问题：

- 具有现实触发条件；
- 明确违反 canonical、accepted contract 或 package invariant；
- 生产路径或真实文档消费者可达；
- 具有用户可感知影响；
- 已检查相反证据；
- 能指出首次允许非法状态的 owner。

以下不能单独成为 finding：

- 风格；
- 命名；
- 纯重构偏好；
- 缺少注释；
- 理论上的 `unwrap` 或 panic；
- 测试覆盖不足；
- future TODO 未实现；
- 离线无法检查在线漏洞；
- 未证实 dead code；
- 没有生产路径或真实文档消费者可达性的猜测。

它们至多进入 verification gap 或 deferred，除非已证明造成当前缺陷或虚假保证。

## 14. 严重性与 Confidence

严重性：

- `P0`：安全边界突破、广泛数据破坏或核心结果系统性不可用。
- `P1`：主要领域结果错误、solver 保证虚假、公开接口严重破坏或高概率崩溃。
- `P2`：具有现实触发条件的局部错误、错误降级、兼容性、持久化或资源问题。
- `P3`：影响有限但仍是可复现的产品缺陷，不包括洁癖或可选 hardening。

Confidence 独立记录：

```text
high
medium
low
```

low-confidence 假设不得进入正式 findings，只能进入待验证项。

## 15. Finding 格式

每条 finding 必须包含：

```text
ID
severity: P0-P3
confidence
category
snapshot_id
title
path:line
symbol
violated invariant
canonical / accepted ADR / contract basis
minimum realistic trigger
production or documentation-consumer reachability chain
actual behavior
expected behavior
user impact
counterexample or reproduction
evidence log/artifact
contrary evidence checked
why existing tests do not disprove it
first illegal-state owner
minimum correct repair boundary
attribution
```

归因只能是 `current defect`，或在有历史证据时使用 `regression relative to <named baseline>`。

其他要求：

- 路径和行号必须对应最终稳定 snapshot。
- 不能只贴搜索命中。
- 同一根因影响多个 consumer 时只报一条，并列出影响面。
- 同一症状若有不同根因，才拆成多条。
- 文档 finding 只有在 current 文档错误声明能力、破坏真源/lifecycle 或误导真实 consumer 时成立。
- 推荐修复只说明最小 owner 边界，不提供补丁。

## 16. 最终 Synthesis 与独立复审

所有可完成 package 结束后，主 Agent 先进行 synthesis：

1. 确保全部结论绑定同一稳定 snapshot。
2. 去重根因。
3. 验证跨 package 调用链。
4. 区分 finding、assurance gap 和 verification gap。
5. 对每个 no-finding surface 反查其 package 必须为 `audited`。
6. 从完整覆盖声明中扣除所有 partial、deferred、blocked、invalidated。
7. 检查 reverse path coverage 是否遗漏 in-scope path。
8. 检查每个 package 的 scope/render gate。
9. 检查所有 shared-proof evidence dependencies。
10. 检查 Documentation Topology Report 与 rewrite handoff：每项确定性判断和改写动作必须追溯到 `audited` package；删除所有依赖 `partial`、`deferred`、`blocked` 或 `invalidated` package 的裁决、建议和 proposed outline，仅保留具名 verification gap 与后续调查要求。
11. 检查 `documentation-topology-synthesis` 的 output safety gate，确认两个 artifacts 仅是 ignored audit evidence，没有承担 current truth、active-change owner 或 tracked replacement 职责。

如果 capability gate 实际可用，调用独立 `sol-reviewer`，向其提供稳定 snapshot ID、reverse path coverage、coverage ledger、所有 findings、rejected candidates、no-finding surfaces、Documentation Topology Report、rewrite handoff、command evidence、scope/render gates 和 drift ledger。

要求 reviewer 只读挑战：

- false positives；
- 遗漏 consumer；
- severity；
- solver/domain guarantee；
- coverage 夸大；
- 快照混用；
- formal 与 non-formal 边界。
- 文档 owner 重复、读者职责混淆和 handoff 中未经证实的重写范围。

reviewer 结论必须绑定同一 snapshot。review 期间发生 drift 时，按全局失效规则处理。

profile 不可用时，只能标记 self-review，不得声称 independent final review，overall `COMPLETE` 被禁止。

## 17. Overall Status Gate

最终状态只能是：

```text
COMPLETE
PARTIAL
BLOCKED
```

`COMPLETE` 只有在以下条件全部满足时允许：

- 最终只有一个稳定 snapshot；
- 每个 in-scope path 至少反向映射到一个 audited package；
- 每个 planned/required package 都是 audited；
- required packages 中不存在 partial、deferred、blocked 或 invalidated；
- 每个 required package 的 scope checker 为 PASS；
- 每个 required package 的 renderer 为 PASS；
- 所有按风险要求的命令 evidence 完整；
- training engine 和 training table 在纳入范围时完成专属协议；
- G19 的 required packages、Documentation Topology Report 和 rewrite handoff 完成且只引用 audited facts；
- `documentation-topology-synthesis` 的 `output_safety_gate = PASS`，且 artifacts 明确标识为 ignored audit evidence、不得承担 current truth 或 active-change owner；
- 没有 unresolved canonical conflict；
- independent final reviewer 实际执行并通过；
- 最终 drift check 通过；
- auditor 没有引入 tracked/nonignored 改动。

否则使用：

- `PARTIAL`：有可用结论，但 coverage、profile、evidence gate、环境或成本导致部分未完成。
- `BLOCKED`：真源冲突、安全违规、审计者写入、无法稳定快照或关键前置条件阻止继续。

即使为 `COMPLETE`，也必须写明：

```text
Complete for the explicitly non-formal whole-repository code-audit scope.
Not a formal named-system acceptance.
```

## 18. 最终输出格式

Findings 必须置顶，按 P0 -> P3 排序。

没有 findings 时，只能写：

```text
在已完成的 audited packages 中未发现达到门槛的问题。
```

不能笼统写“整仓无问题”。

随后按顺序输出：

### 1. Findings

使用第 15 节完整格式。Findings 只输出一次。

### 2. Overall Status

- `COMPLETE | PARTIAL | BLOCKED`
- 是否完成 independent final review
- 明确说明这是 non-formal audit

### 3. Snapshot

- repo
- HEAD
- branch
- snapshot ID
- 起始 status
- 最终 status
- 起始 dirty paths
- 最终 drift
- auditor-introduced tracked/nonignored changes

### 4. Blocked Truth Conflicts

- 冲突文档
- 精确位置
- 受影响 package
- 是否需要 formal-audit 或普通用户裁决

没有则写 `none`。

### 5. Coverage Ledger

每个 group 下列出：

- package ID
- invariant
- analysis status
- primary Skill
- assigned profile
- truth/required/consumer/proof closure
- scope gate
- render gate
- evidence
- findings
- verification gaps

umbrella group 本身不得标 audited。

### 6. Reverse Path Coverage

- in-scope path 总数
- mapped-to-audited 数
- partial/deferred/blocked/invalidated 分状态计数
- explicit exclusions
- 未映射路径

### 7. Documentation Topology Report 与 Rewrite Handoff

报告：

- 文档受众与职责矩阵；
- 事实 owner 矩阵；
- 重复、冲突和易漂事实；
- 人类入口、Agent 路由、领域真源、实现地图、架构解释、API reference、ADR 和历史材料的边界；
- 可生成/可机械校验的表格；
- fresh-agent 导航验收场景；
- `documentation-topology-report.md` artifact；
- `documentation-rewrite-handoff.md` artifact；
- 仅对具有 audited 依据的候选文件报告 `keep | trim | rewrite-in-place | split | generate | unchanged`；其余文件报告 `not adjudicated` 及阻塞 package；
- handoff 中受 partial/deferred/blocked/invalidated 影响的不确定项。

明确写明本轮没有修改 tracked 文档，handoff 需要后续独立 quality writer 执行并重新验证，不得把 artifact 称为当前文档。

### 8. Audited No-Finding Surfaces

只列 analysis status 为 audited 且 stop condition 完成的 package，不得列 partial、deferred、blocked、invalidated。

### 9. Assurance Gaps

只列产品保证本身的证明不足，避免与 findings 和 verification gaps 重复。

### 10. Verification Gaps 与 Deferred

包括：

- profile 不可用；
- 环境限制；
- 外部 repo；
- 未跑类别；
- 未完成入口闭包；
- formal-audit-required；
- checker/renderer 非 PASS；
- 文档拓扑或 rewrite handoff 依赖的未审计事实。

### 11. Evidence

分别报告：

```text
build
targeted-test
full-suite
CLI
performance
format
structure
```

每类给出：

- PASS / FAIL / NOT RUN
- 完整命令
- 输入
- exit code
- 绝对日志路径
- artifact 路径
- snapshot ID
- package ID

解释 current failure 与 baseline regression 的区别。

### 12. Special Routing

报告：

- ordinary system-audit packages；
- formal-audit-required 项；
- training engine maintenance package；
- training table 逐条验收及通过/修改/待裁决统计；
- accepted ADR future boundaries；
- external vulnerability intelligence 状态；
- documentation-quality 后续 writer handoff 状态。

### 13. Read-Only Completion

逐 package 报告：

```text
check_task_scope: PASS | FAIL | UNSATISFIED | NOT_RUN
render_evidence: PASS | FAIL | NOT_RUN
documentation output safety: PASS | FAIL | NOT_RUN
```

然后报告：

```text
docs impact: none, audit-only
commit: none, read-only audit
remaining worktree status: ...
```

结束前最后一次捕获并核对：

```text
git status --short --untracked-files=all
global source manifest
snapshot ID
```

最终措辞必须与 ledger 精确一致。

禁止使用超出证据的措辞：

```text
complete
all paths
globally optimal
no regression
independent
no known vulnerabilities
```

现在开始执行审计。不要先输出泛化计划；完成安全预检、能力门禁和 global snapshot 后，直接建立 inventory、reverse path map、文档事实 owner 矩阵和首批 bounded packages。
